# 54-Shopee 签收破损退款前后端数据库 Redis 实现设计

> 创建日期：2026-05-13  
> 状态：设计完成，尚未实现

## 1. 目标

在 Shopee 虚拟客服对话系统 MVP 中，实现第三个自动触发案例“签收破损退款”。该案例在订单已签收后一段游戏时间内，按订单商品、金额、品类和买家画像触发售后客服会话，由 LLM 扮演买家反馈商品破损、包装损坏或实物无法正常使用，并要求卖家给出处理方案。

本设计覆盖签收破损退款的触发规则、订单/商品上下文快照、客服通用表复用、Redis 触发锁、接口复用、前端展示口径和验收标准。前端样式和布局沿用当前 Chat 抽屉与客服网页版，不新增独立页面，不调整视觉结构。

## 2. 范围与非范围

### 2.1 本期范围

- 扫描已签收订单，按对局游戏时间判断是否处于售后触发窗口。
- 命中后创建 `delivered_damage_refund` 客服会话，绑定 `run_id/user_id/order_id/buyer_profile_id/scenario_code`。
- 会话上下文保存订单、商品、签收时间、商品金额、商品图片、物流/包装相关信息和破损描述快照。
- 买家首条消息由 LLM 生成，表达“刚收到但商品破损/无法使用”的售后诉求。
- 学生发送消息后，后端写入学生消息并调用 LLM 生成买家下一条消息。
- 学生主动结束会话后，后端总结卖家处理方式，输出点评和改进建议并落库。
- Chat 抽屉和 `/shopee/customer-service/web` 复用现有客服会话、消息、发送和结束接口。
- 触发窗口、冷却、消息时间和评分时间全部使用对局游戏时间。

### 2.2 非范围

- 不实现真实退款、退货、补发或赔付状态流转。
- 不让 LLM 直接修改订单、资金、库存、物流或售后状态。
- 不新增真实图片上传/证据审核能力；首版用文字描述和系统已有商品图模拟证据。
- 不接入真实 Shopee Return/Refund API。
- 不改变现有 Chat 抽屉和客服网页版布局。
- 不重做自动回复、快捷回复和模型配置体系。
- 不展示数值分数或满意度等级；首版只展示处理方式归类、点评和建议。

## 3. 现有系统承接点

| 现有模块 | 承接方式 |
| --- | --- |
| 我的订单 | 从 `shopee_orders` 读取签收状态、买家、金额、商品明细和物流信息 |
| 订单商品 | 从订单明细读取商品标题、规格、数量、单价和图片 |
| 物流履约 | 使用 `delivered_at` 或签收物流轨迹作为签收游戏时间依据 |
| 买家池 | 复用订单买家画像，决定情绪强度、耐心和诉求表达 |
| 客服通用表 | 复用 `shopee_customer_service_scenarios/conversations/messages/model_settings` |
| Chat 抽屉 | 展示签收破损退款会话，支持学生回复和结束 |
| 客服网页版 | 展示完整会话、订单/商品上下文、处理方式归类、点评和建议 |
| Redis | 复用客服列表/详情缓存、触发锁、接口限流和写入后缓存失效 |

## 4. 核心流程

```text
订单签收
    ↓
写入 delivered_at 或 delivered 物流轨迹
    ↓
客服触发巡检扫描已签收订单
    ↓
判断当前游戏时间是否处于签收后售后触发窗口
    ↓
检查 run 状态、未处理会话数、同订单去重、每日上限和触发概率
    ↓
创建客服会话 shopee_customer_service_conversations
    ↓
LLM 基于订单、商品和破损上下文生成买家首条售后消息
    ↓
学生在 Chat 抽屉或客服网页版回复
    ↓
后端写入学生消息并生成买家下一条回复
    ↓
学生主动结束会话
    ↓
后端总结处理方式并写入点评建议
```

## 5. 触发规则

### 5.1 触发事件定义

签收破损退款的直接触发事件不是“前端识别到破损图片”，也不是客服系统凭空判断商品一定破损，而是后端在订单签收后生成的售后异常模拟事件：

```text
after_sales_event = delivered_damage_reported
scenario_code = delivered_damage_refund
```

事件生成链路：

```text
真实订单签收
  → 进入签收后售后观察窗口
  → 后端按商品、物流、金额、买家画像计算破损/售后风险
  → 命中概率后生成 delivered_damage_reported 售后异常模拟事件
  → 根据事件快照创建 delivered_damage_refund 客服会话
```

也就是说，“签收”提供事实基础，“售后异常模拟事件”提供破损事实快照，“客服会话”只是该事件的沟通表现。实现时应把触发判断拆成两层：

| 层级 | 判断内容 | 结果 |
| --- | --- | --- |
| 事实层 | 订单是否真实签收，是否处于签收后 2-48 游戏小时窗口 | 决定是否可进入售后候选 |
| 事件层 | 是否按风险模型命中 `delivered_damage_reported` | 决定是否生成破损退款客服会话 |

`delivered_damage_reported` 事件不需要首版单独建表，但必须写入会话 `context_json`，至少包含：事件代码、生成游戏时间、触发风险因子、破损类型、破损严重度、证据提示和主诉商品。这样后续对话不是 LLM 临场编造，而是围绕已落入上下文快照的售后异常事件展开。

### 5.2 与真实订单签收流程的联动

签收破损退款不是独立随机事件，必须由系统内真实订单签收流程驱动：

```text
订单发货
  → 物流轨迹按游戏时间推进
  → 订单写入 delivered_at 或 delivered 物流事件
  → 订单进入 completed/delivered 等签收完成口径
  → 签收后一段游戏时间内进入售后触发候选池
  → 命中风险概率后生成 delivered_damage_reported 事件
  → 根据事件创建签收破损退款会话
```

实现时应把“是否可触发”绑定到订单的签收游戏时间，而不是凭空生成售后咨询。买家反馈的事实依据必须是当前订单已签收，且系统已为该订单生成 `delivered_damage_reported` 售后异常模拟事件。

时间来源必须统一：

| 字段/判断 | 时间来源 |
| --- | --- |
| `delivered_at` | 订单签收时的当前对局游戏时间 |
| delivered 物流事件 `event_time` | 物流节点写入时的当前对局游戏时间 |
| 售后触发起点 | 优先 `delivered_at`，缺失时用 delivered 物流事件 `event_time` |
| 签收后等待时长 | 当前对局游戏时间 - 售后触发起点 |
| 售后有效窗口 | 当前对局游戏时间是否仍在签收后窗口内 |

真实世界时间只允许用于 worker 调度频率；不能用于判断签收后多久触发、是否过期、会话消息时间或评分时间。

### 5.3 触发入口

签收破损退款属于事件型 + 巡检型触发，建议在以下位置接入：

- 自动订单 tick worker 推进物流并生成签收结果后。
- 订单模拟器处理签收/完成状态后。
- 手动物流推进接口把订单推进到 delivered 后。
- 订单列表或订单详情查询时只做兜底检查，不作为主要触发入口，避免查询接口产生过多副作用。

### 5.4 候选订单范围

候选订单必须同时满足：

| 条件 | 口径 |
| --- | --- |
| 订单归属 | `run_id/user_id` 属于当前学生 |
| 对局状态 | run 仍为 `running` |
| 订单状态 | 已签收或已完成，存在 `delivered_at` 或 delivered 物流事件 |
| 发货状态 | 订单曾进入正常发货/运输链路，不是未发货订单 |
| 未取消 | 订单未取消 |
| 未退款 | 首版可仅排除已有售后/退款完成状态；若当前无售后表，则先不触发重复会话 |
| 同订单去重 | 不存在同 `order_id/scenario_code` 的已触发会话 |
| 商品明细 | 至少有一个订单商品可用于上下文和商品卡片展示 |

未发货、运输中、已取消、历史 finished 对局不生成签收破损退款会话。

### 5.5 签收后触发窗口

签收破损反馈应发生在买家收到商品后不久，避免签收很久后仍不断生成售后会话。

| 场景 | 窗口建议 | 买家情绪 |
| --- | --- | --- |
| 刚签收 | 签收后 2-6 游戏小时开始可触发 | 发现问题，语气焦急但仍愿意沟通 |
| 常规售后窗口 | 签收后 6-24 游戏小时 | 明确反馈破损，要求处理方案 |
| 临近过期 | 签收后 24-48 游戏小时 | 情绪更强，担心错过处理时间 |
| 过期窗口 | 超过 48 游戏小时 | 首版不再自动触发新会话 |

触发扫描时若签收未满最短等待时间，暂不创建；若已超过最大窗口，候选失效。

### 5.6 概率与优先级

签收破损退款属于售后高优先级案例，建议基础触发概率 `0.45`，优先级高于物流停滞催单和商品细节追问。

| 条件 | 修正建议 |
| --- | --- |
| 商品金额较高 | `+0.10` |
| 商品数量较多 | `+0.05` |
| 商品标题/类目含易碎、玻璃、陶瓷、灯、镜、瓶、盒等关键词 | `+0.10` |
| 订单跨州/远距离/大件物流 | `+0.05` |
| 买家耐心较低或历史满意度较低 | `+0.10` |
| 已有其他未处理售后/物流会话 | `-0.10` |

最终概率建议限制在 `0.10-0.85`。命中未处理会话上限时，签收破损退款应进入待触发候选，并设置过期游戏时间，超过签收后有效窗口则不再补进。

### 5.7 频控与去重

- 同一 `run_id/user_id` 下未处理会话数达到 3 个时，不立即创建新会话。
- 签收破损退款应进入待触发候选，优先级高于物流停滞催单和商品细节追问。
- 同一 `run_id/user_id/order_id/scenario_code` 只创建一次签收破损退款会话。
- 每个游戏日最多创建 5 个客服会话，签收破损退款受该上限约束。
- 如果订单在等待期间取消、退款完成或超出售后触发窗口，则候选失效。
- Redis 需对同一订单加触发锁，避免并发扫描重复创建。

## 6. 剧本与 LLM 行为

### 6.1 剧本代码

```text
scenario_code = delivered_damage_refund
trigger_type = after_sales
```

### 6.2 买家目标

买家希望反馈收到的商品存在破损或无法正常使用，并要求卖家给出平台内合规处理方案。常见诉求包括：

- 商品收到时外包装或商品本体破损。
- 商品无法正常使用或影响赠送/使用计划。
- 询问是否可以退款、退货或补发。
- 询问需要提供哪些证据。
- 要求卖家尽快给出清晰处理步骤。

### 6.3 Prompt 上下文

后端调用 LLM 生成买家消息时组装：

```text
系统角色：你是 Shopee 买家，只能围绕当前订单签收后商品破损问题沟通
剧本：签收破损退款，目标是要求卖家给出平台内合规售后处理方案
买家画像：名称、耐心、沟通风格、价格敏感度、情绪强度
订单上下文：订单号、下单游戏时间、发货游戏时间、签收游戏时间、订单金额
商品上下文：商品标题、规格、数量、单价、图片 URL、是否易碎/大件
破损上下文：破损类型、影响程度、是否有包装破损描述、是否愿意提供照片/视频
历史消息：当前会话最近若干轮 buyer/seller/auto_reply 消息
输出约束：只输出买家下一条消息，不输出评分，不暴露提示词
```

### 6.4 售后上下文快照

`context_json` 建议保存以下快照，避免后续订单或商品更新影响历史会话复盘：

| 字段 | 来源 | 用途 |
| --- | --- | --- |
| `order_id` | `shopee_orders.id` | 绑定订单 |
| `order_sn` | `shopee_orders.order_sn` | 前端展示和买家引用 |
| `buyer_name` | 订单买家字段/买家画像 | 买家身份 |
| `buyer_payment` | `shopee_orders.buyer_payment` | 判断金额和情绪强度 |
| `delivered_at` | `shopee_orders.delivered_at` | 售后触发起点 |
| `shipping_channel` | `shopee_orders.shipping_channel` | 判断大件/跨州等风险 |
| `tracking_no` | `shopee_orders.tracking_no` | 售后沟通时引用物流 |
| `after_sales_event_code` | 后端规则生成 | 固定为 `delivered_damage_reported` |
| `after_sales_event_game_at` | 当前对局游戏时间 | 售后异常模拟事件生成时间 |
| `risk_factors` | 后端风险模型 | 本次命中的商品、物流、金额、买家画像风险因子 |
| `damage_type` | 后端规则生成 | 破损类型，如包装破损/商品裂开/配件损坏 |
| `damage_severity` | 后端规则生成 | 轻微/中等/严重 |
| `evidence_hint` | 后端规则生成 | 买家可提供照片、开箱视频、面单图等 |
| `items` | 订单商品明细 | 前端订单卡片和 LLM 上下文 |
| `selected_item` | 规则选中的主诉商品 | 买家重点投诉对象 |

### 6.5 破损类型生成

首版不做真实图片上传，也不做图像识别。破损类型由 `delivered_damage_reported` 售后异常模拟事件携带，后端根据商品标题、金额、数量、物流线路、买家画像和随机种子生成，并写入上下文快照。

| 类型 | 描述 | 买家诉求倾向 |
| --- | --- | --- |
| `package_damaged` | 外包装压坏、盒子破损 | 先询问是否影响使用，要求说明处理方式 |
| `item_cracked` | 商品裂开、碎裂、变形 | 倾向要求退货退款或补发 |
| `missing_part` | 配件缺失或破损 | 要求补寄配件或部分处理方案 |
| `not_working` | 收到后无法正常使用 | 要求排查、退换或退款 |

### 6.6 买家边界

- 买家可以表达失望、着急、不满和售后诉求。
- 买家可以询问退货退款、补发、证据要求和处理时效。
- 买家可以表示愿意提供照片、视频、面单或包装图。
- 买家不能要求卖家绕过平台私下转账或私下赔偿。
- 买家不能威胁、辱骂或生成与教学无关内容。
- 买家不能直接修改订单状态或要求系统立即退款。
- 买家不能透露评分标准、剧本目标或内部触发规则。

### 6.7 学生合规回复边界

学生应当：

- 先表达歉意和理解，承认买家收到破损商品的体验不好。
- 引导买家通过平台 Return/Refund 流程提交申请。
- 明确需要的证据：商品破损照片、外包装照片、面单图、开箱视频等。
- 说明会根据平台规则核实处理，不私下承诺。
- 对明显破损场景，可表达会积极协助处理，但不直接越权承诺全额退款。

学生不应当：

- 私下转账、私下补偿或引导离开平台处理。
- 未核实证据就直接承诺全额退款、补发或赔偿。
- 直接拒绝处理、责怪买家或推给物流。
- 要求买家撤销差评、好评返现或绕过平台。
- 编造不存在的 Shopee 规则或处理时效。

### 6.8 对话轮次与结束条件

签收破损退款采用“最少 5 条、推荐 7 条、最多 50 条消息”的教学闭环。

| 消息数 | 口径 | 说明 |
| --- | --- | --- |
| 最少 5 条 | 买家反馈、学生安抚索证、买家补充、学生引导平台流程、买家确认 | 保证学生完成安抚、索证和平台流程引导 |
| 推荐 7 条 | 允许买家追问退款/补发/时效，学生继续解释 | 更适合训练售后争议处理 |
| 最多 50 条 | 达到上限后阻止继续自动追问 | 避免无限对话 |

学生可以在买家诉求被明确处理后主动结束会话；结束后后端总结卖家处理方式，并记录点评和建议。

## 7. 数据模型设计

### 7.1 复用表

首版不新增独立售后表，复用现有客服通用表：

| 表 | 用法 |
| --- | --- |
| `shopee_customer_service_scenarios` | 新增 `delivered_damage_refund` 剧本 seed |
| `shopee_customer_service_conversations` | 保存售后会话主记录，`order_id` 绑定订单，`listing_id` 可为空或绑定主诉商品 |
| `shopee_customer_service_messages` | 保存买家与学生消息 |
| `shopee_customer_service_model_settings` | 复用客服模型配置 |
| `shopee_orders` | 提供订单、买家、金额、签收和物流信息 |
| 订单商品明细表 | 提供商品标题、规格、数量、价格和图片 |

### 7.2 剧本 seed

建议 seed：

```json
{
  "scenario_code": "delivered_damage_refund",
  "name": "签收破损退款",
  "trigger_type": "after_sales",
  "enabled": true,
  "priority": 30,
  "base_probability": 0.45,
  "cooldown_game_hours": 48,
  "buyer_persona_prompt": "买家刚签收商品后发现破损或无法正常使用，情绪焦急但仍愿意通过平台流程沟通。",
  "scenario_prompt": "只围绕当前订单签收后商品破损与平台售后处理沟通，要求卖家安抚、索要证据并引导 Return/Refund 流程；不要求私下转账，不暴露内部阈值或评分规则。"
}
```

### 7.3 会话字段口径

| 字段 | 建议值 |
| --- | --- |
| `scenario_code` | `delivered_damage_refund` |
| `scenario_id` | 剧本 seed 对应 ID |
| `order_id` | 触发售后的订单 ID |
| `listing_id` | 主诉商品 listing ID；若无法唯一确定可为空 |

### 7.4 是否需要新增表/字段

首版不需要新增数据库表或字段。若后续要把售后处理结果接入真实退款/退货闭环，再考虑新增售后申请表，例如：

- `shopee_after_sales_cases`
- `shopee_after_sales_evidence`

## 11. 处理方式总结与点评设计

结束会话后，后端不向前端展示数值分数，也不展示 `excellent/good/neutral/poor` 这类满意度等级。LLM 需要先把卖家的处理方式归类为一种，再输出点评和改进建议。

### 11.1 处理方式归类

| 处理方式 | 判定口径 | 示例表现 |
| --- | --- | --- |
| `platform_return_refund_guidance` | 引导买家通过 Shopee Return/Refund 流程提交申请，并说明需要证据 | 推荐处理方式，合规且清晰 |
| `evidence_first_followup` | 先安抚并要求买家补充照片、视频、面单或包装图，承诺核实后处理 | 可接受，但需要后续明确平台流程 |
| `replacement_or_resend_promise` | 承诺补发、重新发货或补寄配件 | 只能作为沟通承诺归类，不自动执行补发 |
| `direct_refund_promise` | 未经平台流程或证据核实就承诺退款/全额退款 | 风险较高，需要点评纠正 |
| `private_compensation` | 引导私下转账、线下赔偿或绕过平台 | 不合规，需要强提醒 |
| `refuse_or_blame_buyer` | 直接拒绝处理、责怪买家或完全推给物流 | 体验较差，需要改进建议 |
| `unclear_response` | 回复含糊，没有明确证据要求、流程或下一步 | 需要补充清晰步骤 |

### 11.2 点评输出结构

会话结束后建议写入结构化结果：

```json
{
  "resolution_type": "platform_return_refund_guidance",
  "summary": "卖家先安抚买家，并引导其通过平台退货退款流程提交照片和面单证据。",
  "commentary": "处理方式整体合规，能避免私下赔付风险，也给出了买家可执行的下一步。",
  "suggestions": [
    "可以更明确列出需要上传的证据类型，例如商品破损照片、外包装照片和面单图。",
    "可以补充说明会在平台申请提交后及时协助核实。"
  ]
}
```

### 11.3 点评维度

| 维度 | 点评关注点 |
| --- | --- |
| 情绪安抚 | 是否先表达歉意、理解买家收到破损商品的不满 |
| 证据引导 | 是否明确要求照片、视频、面单、包装图等证据 |
| 平台流程合规 | 是否引导 Shopee Return/Refund 流程，避免私下处理 |
| 方案清晰度 | 是否给出下一步、处理路径和买家要做什么 |
| 风险控制 | 是否避免未核实就承诺退款、补发、赔偿或编造规则 |

前端展示时只展示处理方式中文名称、点评和建议，不展示分数。

## 12. 验收标准

### 12.1 触发验收

- 已签收订单在签收后 2-48 游戏小时窗口内可进入候选。
- 未签收、运输中、已取消、历史 finished 对局不触发。
- 同一订单同一 `delivered_damage_refund` 剧本只生成一次会话。
- 未处理会话数达到 3 个时不立即创建，候选可延后且不会超过售后窗口。
- 每游戏日最多 5 个客服会话的上限继续生效。

### 12.2 对话验收

- 买家首条消息围绕签收后破损/无法使用，不偏离订单售后。
- 学生回复后，买家后续追问围绕证据、退款/退货/补发和平台流程。
- 买家不会要求私下转账、绕过平台或直接修改订单状态。
- 学生结束会话后生成处理方式归类、点评和建议，并写入会话详情。

### 12.3 前端验收

- Chat 抽屉能显示签收破损退款会话。
- 客服网页版能显示同一会话和消息。
- 未读红点和右侧 Chat Messages 数量徽标沿用现有已读闭环。
- 商品卡片能展示主诉商品或订单商品信息。
- 历史对局只读，不触发新签收破损退款会话。

### 12.4 数据与缓存验收

- 会话写入后客服列表缓存失效。
- 消息写入、结束评分、标记已读后详情缓存失效。
- Redis 订单级触发锁能避免重复创建。
- `context_json` 保留触发时订单、商品、签收和破损快照。

## 13. 实现顺序建议

1. 新增 `delivered_damage_refund` 剧本 seed。
2. 新增签收破损上下文构建函数。
3. 新增签收订单扫描和触发函数，接入订单 tick/物流签收/订单查询兜底路径。
4. 在买家消息生成函数中增加 `delivered_damage_refund` prompt 分支。
5. 在结束会话逻辑中增加签收破损退款处理方式归类、点评和建议生成分支。
6. 复核 Chat 抽屉和客服网页版会话列表能展示新场景。
7. 运行后端编译与前端构建，使用一笔已签收订单做浏览器联调。

## 14. 风险与取舍

| 风险 | 取舍 |
| --- | --- |
| 没有真实破损图片 | 首版用系统商品图 + 文字破损描述模拟，后续再接 OSS 证据上传 |
| LLM 可能越权承诺退款 | prompt 明确禁止买家要求私下处理，点评中标记越权承诺并给出纠正建议 |
| 售后状态未真实落库 | 首版只做客服训练，不改变资金和订单状态 |
| 候选过期处理复杂 | 可先用 Redis 触发锁和扫描窗口控制，后续再补待触发候选表/队列 |
| 多商品订单主诉商品选择 | 首版按易碎关键词、金额或第一件商品选择主诉商品，并写入上下文快照 |

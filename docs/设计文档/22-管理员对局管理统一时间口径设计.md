# 22-管理员对局管理统一时间口径设计

## 1. 目标
- 为工作台左侧菜单中的“对局管理”建立一套明确、可长期维护的字段口径，避免管理员页面、中间对局列表、右侧统一倒计时中心、后端接口各自使用不同时间语义。
- 明确“真实世界时间”与“游戏内部时间”的映射关系，并规定哪些字段可以直接读数据库，哪些字段必须由后端统一推导。
- 将管理员对局管理中的关键展示字段统一收口到“后端单一权威逻辑”，避免前端继续使用 `manual_end_time || end_time`、旧 `duration_days`、旧 `total_game_days` 等混合口径进行二次推导。
- 为后续继续扩展管理员能力（续期、冻结、强制结束、操作日志、批量运营干预）保留统一的时间语义基础。

## 2. 背景与问题

### 2.1 当前页面位置
- 页面入口位于工作台左侧菜单栏的管理员专区：`对局管理`。
- 页面目标不是独立后台，而是在现有 Setup 管理员菜单体系内承载对局级运营干预能力。
- 当前页面核心关注的是：
  - 查看全部对局的运行状态
  - 理解每一局当前处于真实世界和游戏世界的哪个时间点
  - 对运行中对局执行“延长真实世界时长”的操作

### 2.2 已暴露的问题
在本轮梳理前，管理员对局管理曾出现以下典型问题：
- 中间表格中的 `当前 Day` 在延长后显示异常，例如出现与真实时间进度不匹配的值。
- 右侧统一倒计时中心没有围绕“新的真实结束时间”继续换算，而是仍沿用旧周期或基准周期。
- 部分历史脏数据中，`manual_end_time <= created_at`，导致表格中“创建时间”和“结束时间”显示成同一时刻。
- 前端一度同时混用了：
  - 原始 `duration_days`
  - `manual_end_time`
  - `end_time`
  - 前端本地推导的总周期
  从而导致同一页面不同区域口径不一致。

### 2.3 本文档要解决什么
本文档不再讨论页面布局优化本身，而是重点定义：
- 管理员对局管理每一列到底表示什么
- 每一列应该从哪里来
- 延长操作实际改动的是什么
- “真实世界时长”和“游戏日时长”如何映射
- 后端与前端在这条链路上的职责边界

## 3. 核心时间模型

### 3.1 两套时间概念
系统里必须明确区分以下两套时间：

| 概念 | 含义 | 示例 | 用途 |
|---|---|---|---|
| 真实世界时间 | 管理员和玩家在现实中看到的开始/结束/剩余时间 | 2026-04-21 10:00 开始，2026-04-28 10:00 结束 | 生命周期管理、续期、封盘边界 |
| 游戏内部时间 | 游戏模拟系统中的 Day、游戏日期、游戏时刻 | Day 183、游戏日历中的某个月份、游戏时钟 | 经营模拟推进、阶段联动、历史回溯展示 |

### 3.2 映射规则
管理员对局管理必须遵循统一映射：
- **真实世界 7 天 = 游戏内部 365 天**

由此可推导：
- 真实世界 1 天 ≈ 游戏内部 52.142857 天
- 延长真实世界 7 天 = 新增游戏内部 365 天
- 延长真实世界 14 天 = 新增游戏内部 730 天

### 3.3 单一权威顺序
管理员视图的时间语义必须按如下顺序生成：
1. 先确定真实开始时间 `created_at`
2. 再确定真实结束时间 `effective_end_time`
3. 再得到真实总周期 `real_duration_days`
4. 再由真实总周期映射出总游戏日 `total_game_days`
5. 最后根据当前时刻推导当前 `day_index`

禁止反向使用旧字段推导真实结束时间，例如：
- 不能优先拿旧 `total_game_days` 反推真实周期
- 不能优先拿脏 `duration_days` 直接生成结束时间
- 不能让前端自己在 `manual_end_time` 和 `end_time` 之间做第二套判断

## 4. 页面字段口径定义

管理员“对局管理”中间列表当前字段应按下列方式理解。

### 4.1 对局ID
**含义**
- 这局对局的唯一标识。

**数据来源**
- 数据库主键 / run 主键。

**是否允许前端计算**
- 否。

**说明**
- 纯标识字段，没有派生逻辑。

### 4.2 玩家
**含义**
- 该对局所属玩家账号或用户名。

**数据来源**
- run 关联的用户信息。

**是否允许前端计算**
- 否。

**说明**
- 纯展示字段。

### 4.3 市场
**含义**
- 该对局所属市场或站点。

**数据来源**
- run 的市场字段。

**是否允许前端计算**
- 否。

**说明**
- 纯展示字段。

### 4.4 状态
**含义**
- 当前对局生命周期状态，例如 `running`、`finished`。

**数据来源**
- run 状态字段，必要时由后端在读接口前做结束态收口。

**是否允许前端计算**
- 否。

**说明**
- 前端只负责按状态切换按钮文案或禁用态，不负责重新判定对局是否已经结束。

### 4.5 真实总周期
**含义**
- 从对局真实开始到真实结束，总共持续多少真实世界天数。

**正确语义**
- 它表示“当前这把对局最终持续多久”，不是“基准 7 天”，也不是“最后一次延长的天数”。

**数据来源**
- 必须由后端统一按 `created_at -> effective_end_time` 推导。

**计算口径**
- 初始开局：7 天
- 延长 7 天后：14 天
- 再延长 7 天后：21 天

**是否允许前端计算**
- 不建议。

**原因**
- 一旦前端自己根据原始字段拼装，会重新引入脏 `duration_days`、非法 `manual_end_time`、历史数据不一致等问题。

### 4.6 当前 Day
**含义**
- 当前对局在游戏内部时间轴上推进到第几天。

**正确语义**
- 它不是独立于真实时间轴存在的固定值，而是“当前真实进度映射到总游戏日后得到的结果”。

**数据来源**
- 后端统一推导后返回。

**推导依赖**
- `created_at`
- 当前时刻 `now`
- `effective_end_time`
- `total_game_days`

**是否允许前端计算**
- 管理员对局管理页面不应作为权威口径自行计算。

**说明**
- 右侧统一时间中心若需要展示 Day，可以优先消费后端已返回的 `day_index`；只有在明确的实时显示场景下，才允许围绕同一权威时间轴做前端补充映射，但不得与后端口径冲突。

### 4.7 创建时间
**含义**
- 该对局在真实世界中的开始时间。

**数据来源**
- 数据库 `created_at`。

**是否允许前端计算**
- 否。

**说明**
- 这是所有时间推导的起点。

### 4.8 结束时间
**含义**
- 该对局在真实世界中的最终结束时间。

**正确语义**
- 初始情况下，它等于 `created_at + 初始真实周期`。
- 延长后，它等于“原有效结束时间 + 本次延长的真实天数”。

**数据来源**
- 必须由后端统一 resolver 返回唯一有效值。

**是否允许前端计算**
- 否。

**说明**
- 前端不允许再自行使用 `manual_end_time || end_time` 进行拼接或猜测。
- 前端只展示后端最终返回的 `end_time` / `effective_end_time`。

### 4.9 操作
**含义**
- 对当前对局执行运营干预操作，本期主要是“延长真实世界时长”。

**正确语义**
- 操作改动的不是当前 Day，也不是直接改资金、订单、库存，而是：
  1. 修改真实结束时间
  2. 同步更新真实总周期
  3. 同步更新总游戏日
  4. 让当前 Day 在新的总周期上继续推进

**是否允许前端计算**
- 否。

**说明**
- 前端只负责提交 `extend_days`，并展示后端返回的新结果。

## 5. 延长操作的业务定义

### 5.1 延长的本质
管理员“延长对局”本质上是：
- 给当前对局增加真实世界时长

不是：
- 直接把当前 Day 改大
- 手动覆盖游戏时间
- 重置对局
- 对已结束对局重新开局

### 5.2 延长后的影响链路
若管理员对某局执行 `+7 天`，理论上应发生以下变化：
1. 新的真实结束时间 = 旧的真实结束时间 + 7 天
2. 真实总周期 + 7 天
3. 总游戏日 + 365 天
4. 当前 Day 随后按新的总周期和新的总游戏日继续换算
5. 右侧统一倒计时中心基于新的真实结束时间继续倒计时

### 5.3 延长与续期的状态边界
- 对 `running` 状态对局：使用延长接口
- 对 `finished` 状态对局：如产品允许重新开放，则使用续期/renew 语义；若不允许，则直接拒绝

本期管理员对局管理设计文档默认支持两类操作口径共存，但无论是 extend 还是 renew，都必须回到同一套“真实结束时间 -> 总周期 -> 总游戏日 -> Day”的权威链路。

## 6. 后端职责边界

### 6.1 后端必须负责的事情
后端必须成为管理员对局管理时间口径的唯一事实来源，至少负责：
- 解析有效结束时间
- 过滤非法 `manual_end_time`
- 计算真实总周期
- 计算总游戏日
- 计算当前 Day
- 在管理端列表接口中统一返回以上字段
- 在延长/续期接口中返回新旧对比结果

### 6.2 后端权威字段建议
管理员列表接口建议返回：
- `run_id`
- `username`
- `market`
- `status`
- `created_at`
- `end_time`（统一后的权威结束时间）
- `duration_days`（统一后的真实总周期）
- `base_real_duration_days`
- `base_game_days`
- `total_game_days`
- `day_index`

### 6.3 后端不应该做的事情
- 不应把“原始数据库字段值”和“解析后的有效值”混着返回却不说明语义
- 不应在一个接口里用原始 `duration_days`，另一个接口里又用 resolver 值
- 不应允许前端必须理解脏数据细节才能显示正确结果

## 7. 前端职责边界

### 7.1 前端必须负责的事情
- 展示后端统一返回的字段
- 提交管理员操作请求
- 刷新列表与右侧上下文
- 基于后端字段联动页面展示
- 在 UI 层区分 `running` / `finished` 的操作按钮和提示文案

### 7.2 前端不应该负责的事情
- 不应重新判断哪个结束时间更可信
- 不应重新用旧 `duration_days` 推导真实结束时间
- 不应生成自己的“第二套 Day 计算逻辑”作为管理员表格权威来源
- 不应把 `manual_end_time || end_time` 作为默认展示口径

### 7.3 右侧统一倒计时中心的特殊说明
右侧统一倒计时中心属于“展示层的实时 UI”，允许围绕后端给出的：
- `created_at`
- `effective_end_time`
- `total_game_days`
- `day_index`
做秒级动态展示。

但其前提是：
- 时间轴起点与终点必须沿用后端权威数据
- Day 展示优先吃后端返回 `day_index`
- 不能再引入独立于后端之外的另一套结束时间口径

## 8. 数据口径总表

| 字段 | 含义 | 主要来源 | 是否允许前端自行计算 | 备注 |
|---|---|---|---|---|
| `run_id` | 对局唯一标识 | 数据库主键 | 否 | 纯标识 |
| `username` | 玩家账号 | 用户信息 | 否 | 纯展示 |
| `market` | 市场 | run 字段 | 否 | 纯展示 |
| `status` | 对局状态 | run 状态 / 后端收口 | 否 | 前端只消费 |
| `created_at` | 真实开始时间 | 数据库 | 否 | 时间轴起点 |
| `end_time` | 真实结束时间 | 后端 resolver | 否 | 唯一展示结束时间 |
| `duration_days` | 真实总周期 | 后端统一推导 | 否 | 不盲信脏原始值 |
| `base_real_duration_days` | 基准真实周期 | 配置 / run 基准字段 | 否 | 通常为 7 |
| `base_game_days` | 基准游戏日 | 配置 / run 基准字段 | 否 | 通常为 365 |
| `total_game_days` | 当前总游戏日 | 后端统一推导 | 否 | 与真实总周期同步变化 |
| `day_index` | 当前游戏日 | 后端统一推导 | 管理列表中否 | 前端仅可做展示级补充 |

## 9. 脏数据与回退策略

### 9.1 典型脏数据
管理员对局管理需要考虑历史数据中可能出现：
- `manual_end_time` 为空
- `manual_end_time <= created_at`
- `duration_days` 保留旧错误值
- `total_game_days` 与真实周期不一致

### 9.2 回退原则
统一原则：
- **真实结束时间优先级最高**
- 若 `manual_end_time` 非法，则回退到按有效周期推导结束时间
- 若旧 `duration_days` 明显是脏值，也不能直接拿来给前端展示

### 9.3 管理员页面的容错目标
管理员页面不负责清洗历史库，但要做到：
- 即使数据库里还有历史脏字段，管理员页面也尽量展示出一致、可解释、可运营的结果
- 避免出现“创建时间和结束时间完全一样”这种明显错误
- 避免延长后 Day 反而倒退或异常放大

## 10. 推荐接口语义

### 10.1 管理员列表接口
建议继续由统一读接口返回管理员表格所需全部字段，例如：
- `GET /game/admin/runs/options`

返回值必须保证：
- `duration_days` 是统一解析后的真实总周期
- `end_time` 是统一解析后的真实结束时间
- `day_index` 是围绕同一时间轴生成的当前 Day

### 10.2 管理员延长接口
建议语义：
- `POST /game/admin/runs/{run_id}/extend`

请求体：
```json
{
  "extend_days": 7
}
```

返回建议：
```json
{
  "run_id": 6,
  "old_duration_days": 7,
  "new_duration_days": 14,
  "old_total_game_days": 365,
  "new_total_game_days": 730,
  "new_end_time": "2026-04-28T10:00:00",
  "status": "running"
}
```

### 10.3 管理员续期接口
如果保留已结束对局续期能力，建议单独使用：
- `POST /game/admin/runs/{run_id}/renew`

但其内部也必须沿用同一权威链路，而不是分裂为另一套时间规则。

## 11. 页面展示约束

### 11.1 中间对局列表
中间表格应遵循：
- `真实总周期` 显示统一后的 `duration_days`
- `当前 Day` 显示统一后的 `day_index`
- `结束时间` 只显示后端返回的 `end_time`
- `操作` 只负责提交延长/续期请求，不展示第二套时间语义

### 11.2 选中对局摘要
摘要区域应与表格保持同一口径：
- `总游戏日` 显示 `total_game_days`
- `结束时间` 只显示 `end_time`
- `实际周期` 只显示统一后的 `duration_days`

### 11.3 右侧统一倒计时中心
右侧区域必须与中间表格保持一致：
- 选中的 run 是谁，就围绕谁的 `created_at -> end_time` 建立时间轴
- 当前 Day、剩余倒计时、游戏日期、游戏时刻必须与同一 run 的后端语义一致
- 切换选中 run 后，右侧必须立即切换上下文

## 12. 实施建议

### 12.1 先收口语义，再改展示
推荐改造顺序：
1. 先在后端统一“结束时间 / 真实总周期 / 总游戏日 / Day”口径
2. 再统一管理员列表接口返回值语义
3. 再把前端表格、摘要、右侧时间中心全部改为消费同一组字段
4. 最后才做样式、文案、交互微调

### 12.2 尽量避免新增第二套字段名
若现有字段名已经被前后端广泛使用，优先：
- 保留字段名
- 统一字段值语义

例如：
- `duration_days` 继续保留，但它必须稳定表示“真实总周期”
- `end_time` 继续保留，但它必须稳定表示“有效真实结束时间”

### 12.3 何时可以新增字段
只有在现有字段语义已经无法表达清楚时，才考虑新增更明确字段，例如：
- `effective_end_time`
- `effective_duration_days`

但新增字段前必须评估：
- 是否会造成前端继续混用旧字段和新字段
- 是否会使这套语义更复杂而不是更清晰

## 13. 验收标准

### 13.1 口径验收
- 同一 run 在管理员表格、选中摘要、右侧时间中心中显示的结束时间一致
- 同一 run 在管理员表格和右侧时间中心中使用的 Day 口径一致
- 延长后真实总周期按 7→14→21 天这类真实累计方式增长
- 延长后总游戏日按 365→730→1095 这类映射方式同步增长

### 13.2 异常数据验收
- `manual_end_time` 非法时，管理员页面仍可显示正确结束时间
- 历史脏 `duration_days` 不再直接污染结束时间展示
- 不再出现“创建时间与结束时间相同”的明显异常展示

### 13.3 操作验收
- 对 `running` 对局延长后，表格与右侧时间中心立即刷新为新时间轴
- 对 `finished` 对局若允许续期，则续期后所有时间口径同步更新
- 操作只改变时间相关语义，不直接改资金、订单、库存等经营数据

## 14. 结论
管理员“对局管理”这块的本质，不是简单展示几个字段，而是要把：
- 真实世界开始/结束时间
- 真实总周期
- 总游戏日
- 当前 Day
- 延长操作
统一到一条可解释、可验证、可长期维护的时间链路上。

最终建议是：
- **后端统一生成时间语义**
- **前端只展示同一套权威结果**
- **真实结束时间是所有派生结果的上游锚点**
- **延长操作改的是“真实时间轴”，游戏内部 Day 只是沿该时间轴重新映射的结果**

# 22-单品折扣生效与订单影响设计

## 1. 概述

本文档描述单品折扣活动创建后，在折扣时间段内对买家下单行为的影响机制，包括：
- 折扣价格生效逻辑
- 折扣对下单概率的提升
- 订单记录折扣信息
- 游戏时间与折扣时间段的对应关系

---

## 2. 游戏时间体系

### 2.1 时间轴定义

- **真实时间**：用户所在的真实世界时间（如 2026-04-22 10:00）
- **游戏时间**：游戏内虚拟时间，从"第1天 00:00"开始推进
- **时间比例**：`REAL_SECONDS_PER_GAME_DAY = 604800 / 365 ≈ 1656秒`
  - 约 27.6 分钟真实时间 = 1 游戏天
  - 约 1.15 分钟真实时间 = 1 游戏小时

### 2.2 时间转换

**游戏时间 → 真实时间**
```
真实时间 = run.created_at + (游戏天数 - 1) * REAL_SECONDS_PER_GAME_DAY + 游戏时分秒
```

**示例**
- 用户 04-10 10:00 开局 → `run.created_at = 2026-04-10 10:00:00`
- 游戏第3天 14:30 对应真实时间：
  ```
  2026-04-10 10:00:00 + (3-1) * 1656秒 + 14.5小时 * 69秒/游戏小时
  ≈ 2026-04-10 10:55:12
  ```

### 2.3 折扣时间段的存储

- 前端 `DateTimePicker` 显示格式：`2026-01-03 14:30`（游戏内日期）
- 后端存储：将游戏日期转换为真实世界 datetime 存入 `start_at` / `end_at`
- 判断生效：用 `current_tick`（当前游戏时间对应的真实 datetime）对比

## 3. 折扣生效判断

### 3.1 活动状态

折扣活动状态由 `_resolve_discount_campaign_status()` 判断：

| 状态 | 条件 |
|------|------|
| `draft` | `campaign_status = 'draft'` |
| `disabled` | `campaign_status = 'disabled'` |
| `upcoming` | `current_tick < start_at` |
| `ongoing` | `start_at <= current_tick <= end_at` |
| `expired` | `current_tick > end_at` |

### 3.2 生效条件

订单模拟器生成订单时，折扣生效需满足：
1. 活动状态为 `ongoing`
2. 该 listing 在当前时刻只有一个 ongoing 折扣（创建时已校验冲突）
3. 订单商品的 `listing_id` / `variant_id` 在折扣活动商品列表中

---

## 4. 折扣对订单的影响

### 4.1 成交价变为折扣价

**现状**
```python
# shopee_order_simulator.py line 433
unit_price = int((variant.price if variant else best_listing.price) or 0)
```

**改造后**
```python
# 查询该 listing 是否有 ongoing 的折扣
discount_item = db.query(ShopeeDiscountCampaignItem).join(
    ShopeeDiscountCampaign
).filter(
    ShopeeDiscountCampaign.run_id == run_id,
    ShopeeDiscountCampaign.user_id == user_id,
    ShopeeDiscountCampaignItem.listing_id == best_listing.id,
    ShopeeDiscountCampaignItem.variant_id == (variant.id if variant else None),
    ShopeeDiscountCampaign.start_at <= current_tick,
    ShopeeDiscountCampaign.end_at >= current_tick,
    ShopeeDiscountCampaign.campaign_status.notin_(['draft', 'disabled'])
).first()

if discount_item:
    unit_price = int(discount_item.final_price or 0)
else:
    unit_price = int((variant.price if variant else best_listing.price) or 0)
```

**订单金额计算**
```python
payment = unit_price * quantity
```

折扣价直接影响 `buyer_payment`，买家支付金额降低。

### 4.2 折扣提升下单概率

**现有评分公式**（`shopee_order_simulator.py`）

```python
score = (
    0.32 * category_match
    + 0.22 * price_score
    + 0.14 * quality_score
    + 0.08 * stock_score
    + 0.16 * base_intent
    + 0.08 * impulse
)
order_prob = clamp(0.08 + score * 0.95, 0.05, 0.90)
```

**改造后加入 discount_score**

```python
# discount_score：有 ongoing 折扣时 = discount_percent / 100，无折扣时 = 0
discount_score = (discount_item.discount_value / 100.0) if discount_item else 0.0

score = (
    0.30 * category_match
    + 0.20 * price_score
    + 0.13 * quality_score
    + 0.07 * stock_score
    + 0.15 * base_intent
    + 0.05 * impulse
    + 0.10 * discount_score   # 新增，权重 0.10
)
order_prob = clamp(0.08 + score * 0.95, 0.05, 0.90)
```

**权重调整说明**

| 维度 | 原权重 | 新权重 | 变化 |
|------|--------|--------|------|
| category_match | 0.32 | 0.30 | -0.02 |
| price_score | 0.22 | 0.20 | -0.02 |
| quality_score | 0.14 | 0.13 | -0.01 |
| stock_score | 0.08 | 0.07 | -0.01 |
| base_intent | 0.16 | 0.15 | -0.01 |
| impulse | 0.08 | 0.05 | -0.03 |
| **discount_score** | — | **0.10** | 新增 |
| **合计** | **1.00** | **1.00** | — |

**折扣效果示例**

| 折扣幅度 | discount_score | order_prob 最大增幅 |
|----------|---------------|---------------------|
| 10% | 0.10 | +0.0095 |
| 30% | 0.30 | +0.0285 |
| 50% | 0.50 | +0.0475 |
| 80% | 0.80 | +0.0760 |

### 4.3 订单记录折扣信息

生成订单时，若折扣生效，写入以下字段：

| 字段 | 值 |
|------|----|
| `marketing_campaign_type` | `"discount"` |
| `marketing_campaign_id` | `discount_item.campaign_id` |
| `marketing_campaign_name_snapshot` | `campaign.campaign_name` |

无折扣时这三个字段保持 `null`。

## 5. 前端时间选择器改造

### 5.1 现状

`DateTimePicker` 存储的是浏览器本地真实时间（如 `2026-04-22T14:30`）。

### 5.2 改造方案

**前端**
- `DateTimePicker` 显示格式不变（`YYYY-MM-DD HH:mm`）
- 但初始值由后端 bootstrap 接口返回**游戏当前时间**（格式同为 `YYYY-MM-DD HH:mm`，年份从游戏第1天的年份开始）
- 用户选择的游戏日期提交到后端

**后端**
- `create/bootstrap` 接口在 `form.start_at` / `form.end_at` 中返回游戏时间格式的默认值
- `POST /campaigns` 接口接收游戏时间字符串，调用转换函数将其转为真实 datetime 存库：

```python
def _game_datetime_to_real(game_dt_str: str, run: GameRun) -> datetime:
    """将游戏日期时间字符串转换为真实世界 datetime"""
    game_dt = datetime.strptime(game_dt_str, "%Y-%m-%dT%H:%M")
    # 游戏起始日期（游戏第1天对应的日期）
    game_epoch = datetime(game_dt.year, 1, 1, 0, 0, 0)
    game_elapsed_seconds = (game_dt - game_epoch).total_seconds()
    real_elapsed_seconds = game_elapsed_seconds * (REAL_SECONDS_PER_GAME_DAY / 86400)
    return run.created_at + timedelta(seconds=real_elapsed_seconds)
```

## 6. 数据模型

无需新增表，复用现有字段：

- `ShopeeDiscountCampaign.start_at` / `end_at`：存储转换后的真实 datetime
- `ShopeeDiscountCampaignItem.final_price`：折后价，订单成交价来源
- `ShopeeDiscountCampaignItem.discount_value`：折扣百分比，概率加成来源
- `ShopeeOrder.marketing_campaign_id` / `marketing_campaign_type` / `marketing_campaign_name_snapshot`：记录折扣活动

## 7. 接口变更

| 接口 | 变更内容 |
|------|----------|
| `GET /marketing/discount/create/bootstrap` | `form.start_at` / `end_at` 返回游戏时间格式默认值 |
| `POST /marketing/discount/campaigns` | 接收游戏时间，后端转换为真实 datetime 存库 |
| `simulate_orders_for_run` | 新增折扣查询、折扣价成交、概率加成、订单折扣字段写入 |

---

## 8. 验收标准

1. 折扣活动 ongoing 期间，买家下单成交价为折后价，`buyer_payment` 正确
2. 折扣活动 ongoing 期间，`order_prob` 相比无折扣时有可见提升
3. 折扣活动 upcoming / expired 期间，成交价和概率均不受影响
4. 生成的订单记录中 `marketing_campaign_id` 正确写入
5. 折扣创建页时间选择器显示游戏时间，存库后判断逻辑正确

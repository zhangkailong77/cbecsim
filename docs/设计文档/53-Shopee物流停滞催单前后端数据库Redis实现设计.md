# 53-Shopee 物流停滞催单前后端数据库 Redis 实现设计

> 创建日期：2026-05-13  
> 状态：后端 V1 已实现

## 1. 目标

在 Shopee 虚拟客服对话系统 MVP 中，实现第二个自动触发案例“物流停滞催单”。该案例在订单已发货且物流轨迹长时间未推进时，生成买家催单会话，由 LLM 扮演买家围绕包裹不动、预计送达和卖家跟进进行追问，学生在现有 Chat 抽屉或客服网页版中处理。

本设计只覆盖物流停滞催单的后端触发规则、数据库字段复用、Redis 缓存、客服接口复用和前端数据接入口径。前端样式和布局沿用当前已完成页面，不调整视觉、间距、宽度或组件结构。

## 2. 范围与非范围

### 2.1 本期范围

- 扫描运输中订单，按对局游戏时间判断物流轨迹是否停滞。
- 命中后创建 `logistics_stalled_urge` 客服会话，绑定 `run_id/user_id/order_id/buyer_profile_id/scenario_code`。
- 会话上下文保存订单、商品、物流渠道、运单号、ETA、最后物流轨迹和停滞时长快照。
- 买家首条消息由 LLM 生成，表达包裹长时间不动的焦虑和催促。
- 学生发送消息后，后端写入学生消息并调用 LLM 生成买家下一条消息。
- 学生主动结束会话后，后端生成满意度评分并落库。
- Chat 抽屉和 `/shopee/customer-service/web` 共用现有客服会话、消息、发送和结束接口。
- 全部触发、停滞阈值、消息时间和评分时间使用对局游戏时间。

### 2.2 非范围

- 不修改前端样式和布局。
- 不新增独立客服接口路径；优先复用现有 customer-service 接口。
- 不让 LLM 修改订单、发货、签收、退款或取消状态。
- 不自动承诺退款、补发、取消订单或赔偿。
- 不接入真实 Shopee Chat API 或真实物流 API。
- 不实现签收破损退款案例。
- 不重做自动回复、快捷回复和模型配置体系。
- 不立即把满意度反向接入物流履约或订单取消概率；首版先落库和展示。

## 3. 现有系统承接点

| 现有模块 | 承接方式 |
| --- | --- |
| 我的订单 | 从 `shopee_orders` 读取订单状态、买家、金额、商品明细和发货信息 |
| 物流履约 | 复用发货、物流轨迹、ETA、运输中状态和手动推进接口的数据 |
| 订单模拟 | 在游戏 tick 推进订单与物流时触发停滞巡检 |
| 买家池 | 复用订单买家画像，生成催单语气、耐心和情绪强度 |

| 客服网页版 | 展示完整会话、订单/物流上下文和满意度结果 |
| Redis | 复用客服列表/详情缓存、触发锁、接口限流和写入后缓存失效 |

## 4. 核心流程

```text
订单发货，进入运输中
      ↓
物流轨迹按游戏时间推进
      ↓
客服触发巡检扫描 shipping 订单
      ↓
判断最后物流轨迹距当前游戏时间是否超过停滞阈值
      ↓
检查 run 状态、未处理会话数、同订单去重、每日上限、ETA 情绪等级
      ↓
创建客服会话 shopee_customer_service_conversations
      ↓
LLM 基于订单和物流上下文生成买家首条催单消息
      ↓
学生在 Chat 抽屉或客服网页版回复
      ↓
后端写入学生消息并生成买家下一条回复
      ↓
学生主动结束会话
      ↓
后端评分并写入满意度
```

## 5. 触发规则

### 5.1 与真实运输流程的联动

物流停滞催单不是独立随机事件，必须由系统内真实订单运输流程驱动：

```text
学生发货
  → 订单写入 shipped_at、eta_start_at、eta_end_at、tracking_no、waybill_no
  → 订单进入 type_bucket='shipping'
  → 物流轨迹表写入首个 event_time
  → 后续订单 tick 按游戏时间尝试推进物流节点
  → 只有当运输中订单的物流节点长时间未推进时，才进入催单判断
```

实现时应把“是否停滞”绑定到 `ShopeeOrderLogisticsEvent.event_time` 的推进状态，而不是额外按概率凭空生成催单。换句话说，买家催单的事实依据必须是当前订单确实处于运输中，且系统记录的最后一条物流轨迹已经按游戏时间停留超过阈值。

时间来源必须统一：

| 字段/判断 | 时间来源 |
| --- | --- |
| `shipped_at` | 学生发货时的当前对局游戏时间 |
| `eta_start_at` / `eta_end_at` | 发货游戏时间 + 物流渠道/距离换算出的游戏运输时长 |
| `event_time` | 物流节点写入时的当前对局游戏时间 |
| 停滞时长 | 当前对局游戏时间 - 最后一条物流轨迹 `event_time` |
| 是否超 ETA | 当前对局游戏时间 > `eta_end_at` |

真实世界时间只允许用于 worker 调度频率，例如“服务多久醒来跑一次 tick”；不能用于判断订单是否停滞、是否超 ETA、是否触发客服会话。

### 5.2 触发入口

物流停滞催单属于巡检型触发，建议在以下位置接入：

- 自动订单 tick worker 推进游戏时间后。
- 订单模拟器处理运输中订单后。
- 手动物流推进接口完成后，可顺带重新判断当前订单是否仍需催单候选。
- 客服会话列表查询时只做兜底检查，不作为主要触发入口，避免查询接口产生过多副作用。

### 5.3 候选订单范围

候选订单必须同时满足：

| 条件 | 口径 |
| --- | --- |
| 订单归属 | `run_id/user_id` 属于当前学生 |
| 对局状态 | run 仍为 `running` |
| 订单状态 | `type_bucket='shipping'` |
| 处理状态 | `process_status='processed'` 或等价已发货口径 |
| 发货时间 | `shipped_at` 不为空 |
| 物流单号 | `tracking_no` 或 `waybill_no` 至少一个存在 |
| 未签收 | `delivered_at` 为空 |
| 未取消 | `type_bucket!='cancelled'` |
| 同订单去重 | 不存在同 `order_id/scenario_code` 的未结束或已触发会话 |

已完成、已取消、未发货、待付款和历史 finished 对局不生成物流停滞催单。

### 5.4 停滞阈值

停滞判断以最后一条物流轨迹 `ShopeeOrderLogisticsEvent.event_time` 为准；没有物流轨迹时以 `shipped_at` 作为兜底起点。

| 场景 | 阈值建议 | 买家情绪 |
| --- | --- | --- |
| ETA 开始前 | 最后轨迹超过 1.5 游戏日未更新 | 轻度焦虑，询问是否正常 |
| ETA 窗口内 | 最后轨迹超过 1 游戏日未更新 | 中度焦虑，要求解释和跟进 |
| 超过 ETA 结束 | 当前游戏时间晚于 `eta_end_at` 且最后轨迹超过 0.5 游戏日未更新 | 强烈焦虑，要求明确处理方案 |
| 长距离/东马线路 | 阈值可额外放宽 0.5 游戏日 | 允许跨州或东马物流更慢 |
| 大件物流 | 阈值可额外放宽 0.5 游戏日 | 大件配送节点更少 |

所有阈值均按对局游戏时间计算，不使用服务器真实当前时间。

### 5.5 概率与优先级

物流停滞催单比商品细节追问优先级更高，建议基础触发概率 `0.55`。

| 条件 | 修正建议 |
| --- | --- |
| 当前游戏时间已超过 `eta_end_at` | `+0.25` |
| 商品金额较高 | `+0.10` |
| 买家耐心较低 | `+0.10` |
| 同买家历史有低满意客服记录 | `+0.05` |
| 物流渠道为长距离或大件 | `-0.05` |
| 已有其他未处理物流/售后会话 | `-0.10` |

最终概率建议限制在 `0.15-0.90`。超过 ETA 的高风险订单即使命中未处理会话上限，也应进入待触发候选，待学生处理完一个会话后优先补进。

### 5.6 频控与去重

- 同一 `run_id/user_id` 下未处理会话数达到 3 个时，不立即创建新会话。
- 物流停滞催单不直接跳过，应记录待触发候选或通过 Redis 触发锁延后重试。
- 同一 `run_id/user_id/order_id/scenario_code` 只创建一次物流停滞催单会话。
- 每个游戏日最多创建 5 个客服会话，物流停滞催单受该上限约束。
- 如果订单在等待期间已签收、取消或进入 completed/cancelled，则候选失效。
- 如果最后物流轨迹已更新且不再满足停滞阈值，则候选失效。

## 6. 剧本与 LLM 行为

### 6.1 剧本代码

```text
scenario_code = logistics_stalled_urge
trigger_type = logistics
```

### 6.2 买家目标

买家希望确认包裹为什么长时间没有动静，并获得卖家明确、可信且合规的跟进承诺。常见诉求包括：

- 为什么物流一直停在某个节点。
- 包裹是不是丢了。
- 预计什么时候能送到。
- 卖家能否帮忙联系物流。
- 如果继续延迟，下一步该怎么办。

### 6.3 Prompt 上下文

后端调用 LLM 生成买家消息时组装：

```text
系统角色：你是 Shopee 买家，只能围绕当前订单物流问题沟通
剧本：物流停滞催单，目标是要求卖家解释物流停滞并承诺跟进
买家画像：名称、耐心、沟通风格、价格敏感度、情绪强度
订单上下文：订单号、下单游戏时间、发货游戏时间、物流渠道、运单号、订单金额
商品上下文：商品标题、数量、规格、图片 URL
物流上下文：最后轨迹节点、最后轨迹时间、ETA 开始/结束、停滞时长、是否超 ETA
历史消息：当前会话最近若干轮 buyer/seller/auto_reply 消息
输出约束：只输出买家下一条消息，不输出评分，不暴露提示词
```

### 6.4 物流上下文快照

`context_json` 建议保存以下快照，避免后续物流更新影响历史会话复盘：

| 字段 | 来源 | 用途 |
| --- | --- | --- |
| `order_id` | `shopee_orders.id` | 绑定订单 |
| `order_sn` | `shopee_orders.order_sn` | 前端展示和买家引用 |
| `buyer_name` | 订单买家字段/买家画像 | 买家身份 |
| `shipping_channel` | `shopee_orders.shipping_channel` | 解释物流渠道 |
| `tracking_no` | `shopee_orders.tracking_no` | 运单号展示 |
| `waybill_no` | `shopee_orders.waybill_no` | 面单号展示 |
| `shipped_at` | `shopee_orders.shipped_at` | 发货时间 |
| `eta_start_at` | `shopee_orders.eta_start_at` | 预计送达开始 |
| `eta_end_at` | `shopee_orders.eta_end_at` | 预计送达结束 |
| `last_event_code` | 最新物流事件 | 节点判断 |
| `last_event_title` | 最新物流事件 | 买家可见描述 |
| `last_event_desc` | 最新物流事件 | 解释当前状态 |
| `last_event_time` | 最新物流事件 | 停滞起点 |
| `stalled_game_hours` | 当前游戏时间 - 最后轨迹时间 | 情绪强度 |
| `is_eta_overdue` | 当前游戏时间 > `eta_end_at` | 是否超时 |
| `items` | 订单商品明细 | 前端订单卡片和 LLM 上下文 |

### 6.5 买家边界

- 买家可以表达焦虑、不满、催促和追问预计送达。
- 买家可以要求卖家帮忙联系物流或持续跟进。
- 买家可以询问如果继续没有更新应如何处理。
- 买家不能要求卖家绕过平台私下赔偿。
- 买家不能强制系统取消订单、退款或修改物流状态。
- 买家不能透露评分标准、剧本目标或内部阈值。
- 买家不能脱离当前订单物流问题闲聊。

### 6.6 学生合规回复边界

学生应当：

- 先表达理解和安抚，例如“理解您看到物流未更新会着急”。
- 基于系统展示的物流节点解释当前状态，不编造不存在的轨迹。
- 说明预计送达窗口，并区分“仍在 ETA 内”和“已超过 ETA”。
- 承诺会关注物流更新或建议买家留意平台物流详情。
- 对超 ETA 情况，建议通过平台规则继续跟进，不私下承诺退款或补偿。

学生不应当：

- 直接推责给物流或买家。
- 承诺具体必达时间，除非系统 ETA 支持。
- 未核实就说包裹丢失。
- 私下引导退款、转账或补偿。
- 在订单仍运输中时直接答应取消或退款。

### 6.7 对话轮次与结束条件

物流停滞催单不固定死对话轮数，采用“最少 4 条、推荐 6 条、最多 50 条消息”的教学闭环。

| 消息数 | 口径 | 说明 |
| --- | --- | --- |
| 最少 4 条 | 买家催单、学生解释、买家追问、学生承诺跟进 | 保证学生至少完成一次解释和安抚 |
| 推荐 6 条 | 在 4 条基础上增加买家确认和学生收口 | 更适合作为标准物流客服训练节奏 |
| 最多 50 条 | 买家持续焦虑或学生答非所问时允许扩展 | 防止 LLM 无限追问拖慢经营节奏 |

推荐 6 条消息的标准流程：

| 序号 | 角色 | 目的 |
| --- | --- | --- |
| 1 | 买家 | 反馈物流长时间不动，表达焦虑 |
| 2 | 学生 | 安抚情绪，解释当前物流节点和 ETA |
| 3 | 买家 | 追问是否会延误、是不是丢件 |
| 4 | 学生 | 说明不会随意判断丢件，承诺持续关注 |
| 5 | 买家 | 要求如果继续不动该怎么办 |
| 6 | 学生 | 给出平台内后续处理路径并礼貌收口 |

结束规则：

- 少于 4 条消息时允许学生点击结束，但评分应识别为可能过早结束。
- 达到 6 条消息且买家目标已满足时，LLM 应进入收口状态，不继续制造新问题。
- 达到 50 条消息后，系统应阻止继续自动生成买家追问，并提示学生结束会话或做最后补充。
- 会话结束后进入满意度评分，不再生成新的买家消息。

## 7. 满意度评分

学生主动结束会话后，后端基于会话消息、订单和物流上下文生成评分。

| 维度 | 分值 | 物流停滞催单评分重点 |
| --- | --- | --- |
| 情绪安抚 | 0-25 | 是否先共情买家焦虑，语气礼貌耐心 |
| 物流解释准确性 | 0-25 | 是否基于真实物流节点、ETA 和停滞时长解释 |
| 跟进承诺 | 0-20 | 是否给出合理跟进方式，不空泛敷衍 |
| 平台合规 | 0-20 | 是否未私下承诺退款、补偿、取消或虚假时效 |
| 订单稳定 | 0-10 | 是否降低买家取消/投诉风险 |

满意度等级：

| 等级 | 分数区间 | 处理 |
| --- | --- | --- |
| `high` | 85-100 | 记录高满意，可后续降低该买家取消/投诉倾向 |
| `medium` | 60-84 | 记录正常完成 |
| `low` | 0-59 | 记录低满意，可后续提高取消、退款或差评风险参考 |

首版只落库和展示评分，不立即影响物流推进、订单状态或取消概率。

## 8. 数据库设计

本期优先复用 51/52 号设计中已经落地的客服通用表，不新增业务表。物流停滞候选首版可通过订单巡检即时判断；若后续需要跨未处理会话上限持久排队，再补充候选表或 Redis 延迟队列。

### 8.1 `shopee_customer_service_scenarios`

新增或 seed：

| 字段 | 值 |
| --- | --- |
| `scenario_code` | `logistics_stalled_urge` |
| `name` | `物流停滞催单` |
| `trigger_type` | `logistics` |
| `enabled` | `true` |
| `base_probability` | `0.55` |
| `cooldown_game_hours` | `48` |
| `buyer_persona_prompt` | 物流催单买家画像提示词 |
| `scenario_prompt` | 物流停滞催单目标和边界 |
| `rubric_json` | 本文第 7 节评分规则 |

### 8.2 `shopee_customer_service_conversations`

物流停滞催单会话要求：

- `scenario_code='logistics_stalled_urge'`。
- `order_id` 必填。
- `listing_id` 可填订单中的主商品 ID；多商品订单可取最高金额商品或首个订单商品作为摘要展示。
- `buyer_profile_id` 复用订单买家画像。
- `trigger_reason` 记录停滞原因，例如“物流 32 游戏小时未更新，已超过预计送达结束时间”。
- `context_json` 保存订单和物流快照。
- `status` 初始为 `open`。

### 8.3 `shopee_customer_service_messages`

继续复用现有消息表：

- 买家首条消息：`sender_type='buyer'`，`message_type='text'`。
- 学生回复：`sender_type='seller'`。
- 后续买家追问：`sender_type='buyer'`。
- `sent_game_at` 使用当前对局游戏时间。

### 8.4 现有订单和物流字段复用

| 表/模型 | 字段 | 用途 |
| --- | --- | --- |
| `shopee_orders` | `type_bucket` | 判断是否为 `shipping` |
| `shopee_orders` | `process_status` | 判断是否已处理/已发货 |
| `shopee_orders` | `shipped_at` | 发货时间和停滞兜底起点 |
| `shopee_orders` | `delivered_at` | 已签收则不触发 |
| `shopee_orders` | `eta_start_at` / `eta_end_at` | ETA 情绪强度和超时判断 |
| `shopee_orders` | `tracking_no` / `waybill_no` | 物流单号展示 |
| `shopee_orders` | `shipping_channel` | 渠道解释和阈值修正 |
| `shopee_orders` | `distance_km` | 长距离阈值修正 |
| `shopee_order_logistics_events` | `event_code` | 节点编码 |
| `shopee_order_logistics_events` | `event_title` / `event_desc` | 前端展示和 LLM 解释 |
| `shopee_order_logistics_events` | `event_time` | 停滞时长计算 |

## 9. Redis 与缓存设计

### 9.1 复用缓存

- 客服会话列表缓存：创建会话、写入消息、结束会话后失效。
- 客服会话详情缓存：写入消息和结束会话后失效。
- 订单列表缓存：物流状态变化或新增客服触发状态摘要后按现有订单缓存前缀失效。
- LLM 调用限流：复用客服模型调用限流。

### 9.2 触发锁

建议新增 Redis key：

```text
shopee:customer-service:logistics-stalled:lock:{run_id}:{order_id}
```

用途：

- 防止同一个订单在同一游戏 tick 或并发请求中重复创建会话。
- TTL 建议为 48 游戏小时对应的真实秒数，或至少覆盖一次自动 tick 周期。
- 创建会话成功后，可保留锁直到订单签收/取消或会话结束后冷却期结束。

### 9.3 待触发候选

首版可不新增持久表，采用巡检再次判断即可。若需要保留因未处理会话上限被阻塞的高优先级催单，建议 Redis sorted set：

```text
shopee:customer-service:pending:{run_id}:logistics_stalled_urge
```

- score：候选优先级时间或风险分。
- value：`order_id`。
- 失效条件：订单签收、取消、物流轨迹更新后不再停滞、超过 ETA 后太久仍未处理。

### 11.1 Chat 抽屉

- 会话标题仍展示买家名。
- 摘要消息展示买家最后一句催单内容。
- 点击会话后打开现有详情浮窗。

### 11.2 客服网页版

- 左侧会话列表复用现有客服列表。
- 中间消息流复用现有聊天组件。
- 右侧商品/上下文区域后续可展示订单物流卡片：订单号、物流渠道、运单号、最后轨迹、ETA 和停滞时长。
- 本设计不调整页面宽度、颜色、菜单栏、顶部导航或已有组件结构。

### 11.3 输入与结束

- 输入框启用条件继续依赖 `can_send`。
- 单会话最多 50 条消息。
- 结束会话按钮继续走现有 resolve 接口。
- 历史对局继续只读。

## 12. 游戏时间口径

物流停滞催单的业务判断必须以对局游戏时间为唯一口径。真实时间只决定后台 worker 多久运行一次，不参与任何业务阈值计算。

| 场景 | 时间来源 |
| --- | --- |
| 停滞判断 | 当前对局游戏时间 - 最后物流轨迹游戏时间 |
| ETA 判断 | `eta_start_at/eta_end_at` 与当前对局游戏时间比较 |
| 买家首条消息时间 | 当前对局游戏时间 |
| 学生回复时间 | 当前对局游戏时间 |
| 满意度评分时间 | 当前对局游戏时间 |
| 每日会话上限 | 对局游戏日 |

禁止使用浏览器真实时间或服务器真实当前时间作为业务判断来源。实现时不得使用 `datetime.now()`、浏览器 `Date.now()` 或请求到达时间判断停滞；只能使用 `_resolve_game_tick(...)` 得到的当前对局游戏时间与订单/物流表中已经保存的游戏时间字段比较。

## 13. 实现步骤建议

1. 后端 seed `logistics_stalled_urge` 剧本。
2. 增加物流停滞候选扫描函数，复用订单、物流轨迹和当前游戏时间。
3. 在订单 tick/物流推进后调用扫描函数。
4. 创建会话时写入订单与物流快照 `context_json`。
5. 扩展 LLM prompt builder，按 `scenario_code` 组装物流催单上下文。
6. 扩展满意度评分 prompt 和 rubric。
7. 保持 Chat 抽屉与客服网页版现有样式，只接入物流上下文字段展示。
8. 补充 Redis 触发锁和缓存失效。
9. 增加后端编译、核心触发单元测试或手动数据验证。

## 14. 验收标准

- 物流停滞催单只能由真实运输流程触发：订单已发货、进入运输中、存在发货时间/物流轨迹，且物流轨迹确实长时间未推进。
- 运输中订单在最后物流轨迹超过停滞阈值后，可按概率生成 `logistics_stalled_urge` 会话。
- 同一订单不会重复生成物流停滞催单会话。
- 未处理会话达到 3 个时，物流催单不会丢失高风险候选，后续可补触发。
- 已签收、已取消、未发货或历史对局订单不会触发。
- 买家首条消息围绕包裹不动、ETA 和跟进诉求，不脱离订单物流上下文。
- 停滞时长和 ETA 超时判断必须使用当前对局游戏时间与 `event_time/eta_end_at` 比较，不得使用真实世界时间。
- 学生回复后，系统可继续生成买家追问，消息时间使用对局游戏时间。
- 结束会话后生成满意度评分，评分重点覆盖情绪安抚、物流解释准确性、跟进承诺、平台合规和订单稳定。
- Chat 抽屉和客服网页版能读取同一会话和消息数据。
- 不新增前端视觉风格，不改变现有客服入口和布局。

## 15. 风险与后续

- 如果物流轨迹推进过于频繁，停滞场景难以触发，需要在模拟器中保留一定概率的物流节点延迟。
- 如果停滞阈值过短，会导致买家催单过多；阈值应结合 ETA、距离和物流渠道动态调整。
- 如果未处理会话上限长期满载，物流催单候选需要 Redis 或数据库候选队列保证不丢失。
- 后续签收破损退款案例实现后，应明确三个客服案例的优先级和同订单互斥规则。

# 27 - 套餐优惠（Bundle）概率计算设计

> 创建日期：2026-04-27
> 状态：设计完成，待实现

## 目标

将套餐优惠（Bundle Campaign）接入订单模拟器，使已创建的 bundle 活动在买家下单时真实生效，影响购买数量和成交价。

## 背景

- 单品折扣已通过 `effective_price → price_score → best_score → order_prob` 链路接入订单模拟（见 23、24 号设计文档）
- 套餐优惠已实现创建页、后端接口、数据库写入、Redis 缓存（见首次对话追踪结果），但**未进入订单模拟**
- 当前模拟器每个买家每 tick 最多下 1 件，套餐优惠需要支持多件购买

## 核心差异：套餐 vs 单品折扣

| 维度 | 单品折扣 | 套餐优惠 |
|---|---|---|
| 影响对象 | 单价 | 购买数量 + 单价 |
| 价格变化 | 直接替换成交价 | 按阶梯折算每件单价 |
| 数量 | 固定 1 件 | 按阶梯 2/3/... 件 |
| 模拟器改动 | 替换 price | 新增加购升级步骤 |

## 流程设计

### 现有流程（不变）

```
买家激活 → 候选商品评分 → 选出 best_listing → 选变体 → 算 order_prob → 下单(qty=1)
```

### 新增步骤：Bundle Upgrade

```
买家激活 → 候选商品评分 → 选出 best_listing → 选变体
  → 查 bundle_map
  → 有 bundle？
    → 是：计算加购吸引力 → 逐级衰减滚阶梯 → 确定 qty 和 unit_price
    → 否：保持 qty=1，原价
  → 算 order_prob → 下单(qty=N)
```

**设计原则：在现有选品逻辑之后插入 bundle upgrade 环节，不改变已有的单品折扣逻辑。**

## 三种 Bundle 类型的统一处理

### 类型定义

| 类型 | `bundle_type` | 含义 | 举例 |
|---|---|---|---|
| 折扣比例 | `percent` | 买 N 件，享 X% 折扣 | 买2件享10% off |
| 固定金额减免 | `fixed_amount` | 买 N 件，立减 RM X | 买2件立减8 RM |
| 套餐价 | `bundle_price` | 买 N 件，套餐价 RM X | 买2件套餐价85 RM |

### 每件折后单价计算

三种类型只在计算 `effective_unit_price` 时不同，概率公式完全共用：

```python
def _resolve_bundle_unit_price(
    *, bundle_type: str, original_price: float, tier: dict
) -> float:
    qty = tier["buy_quantity"]
    value = float(tier["discount_value"])

    if bundle_type == "percent":
        # 折扣比例：直接按百分比折扣
        return original_price * (1 - value / 100)

    elif bundle_type == "fixed_amount":
        # 固定金额减免：总价减去固定金额后平摊到每件
        return max(1, (original_price * qty - value) / qty)

    elif bundle_type == "bundle_price":
        # 套餐价：直接用套餐价除以件数
        return max(1, value / qty)

    return original_price
```

### 计算举例（原价 50 RM）

| 类型 | 阶梯 | 参数 | 每件折后单价 | 节省比例 |
|---|---|---|---|---|
| `percent` | 买2件享10% off | value=10 | 50 × 0.9 = 45.0 | 10% |
| `fixed_amount` | 买2件立减8 RM | value=8 | (100-8)/2 = 46.0 | 8% |
| `bundle_price` | 买2件套餐价85 RM | value=85 | 85/2 = 42.5 | 15% |

## 概率计算公式

### 1. 加购吸引力（统一公式）

原始 `savings_rate × price_sensitivity × impulse_level` 过于保守：例如节省 10%、价格敏感度 0.5、冲动系数 0.5 时，触发概率只有 2.5%，套餐几乎不会命中。因此采用带基础兴趣项的统一公式：

```python
# 每件节省比例，负优惠不产生吸引力
savings_rate = clamp((original_price - effective_unit_price) / original_price, 0.0, 0.95)

# 购买力约束（N 件总价不能超出买家承受范围太多）
buyer_budget = 30 + buyer.purchase_power * 300
afford_factor = clamp(buyer_budget / (effective_unit_price * tier_qty), 0.60, 1.0)

# 加购概率：基础兴趣 + 节省吸引 + 冲动加成，再乘购买力约束
upgrade_prob = clamp(
    0.16
    + savings_rate * (0.9 + buyer.price_sensitivity * 0.7)
    + buyer.impulse_level * 0.12,
    0.0,
    0.85,
) * afford_factor
```

### 2. 阶梯选择（逐级衰减）

从最低阶梯开始，逐级判断是否升级：

```python
available_tiers = [
    tier for tier in sorted(bundle_tiers, key=lambda t: t["buy_quantity"])
    if tier["buy_quantity"] <= sellable_stock
]

selected_qty = 1
selected_unit_price = original_price
selected_savings_rate = 0.0

for tier in available_tiers:
    tier_qty = tier["buy_quantity"]
    unit_price = _resolve_bundle_unit_price(
        bundle_type=bundle_type,
        original_price=original_price,
        tier=tier,
    )
    savings_rate = clamp((original_price - unit_price) / original_price, 0.0, 0.95)
    afford_factor = clamp(buyer_budget / (unit_price * tier_qty), 0.60, 1.0)
    upgrade_prob = clamp(
        0.16
        + savings_rate * (0.9 + buyer.price_sensitivity * 0.7)
        + buyer.impulse_level * 0.12,
        0.0,
        0.85,
    ) * afford_factor

    if rng.random() > upgrade_prob:
        break

    selected_qty = tier_qty
    selected_unit_price = unit_price
    selected_savings_rate = savings_rate
```

**衰减逻辑说明：**
- 先决定要不要从 1 件升到 2 件（最低阶梯）
- 再决定要不要从 2 件升到 3 件
- 越高的阶梯越难触发，更自然地模拟真实买家行为

### 3. 下单概率调整

由于 bundle upgrade 发生在选出 `best_listing` 之后，必须用 `selected_unit_price` 重算一次 `price_score` 和候选总分，否则套餐优惠不会真正通过价格分影响 `order_prob`。

```python
base_order_prob = clamp(0.08 + best_score * 0.95, 0.05, 0.90)

bundle_price_score, _ = _resolve_price_score(
    price=selected_unit_price,
    target_price=30 + buyer.purchase_power * 300,
    price_sensitivity=buyer.price_sensitivity,
)

bundle_score = (
    0.32 * category_match
    + 0.22 * bundle_price_score
    + 0.14 * quality_score
    + 0.08 * stock_score
    + 0.16 * base_intent
    + 0.08 * impulse
)

if bundle_applied:
    bundle_bonus = clamp(selected_savings_rate * 0.08, 0.0, 0.06)
    bundle_order_prob = clamp(0.08 + bundle_score * 0.95 + bundle_bonus, 0.05, 0.90)
    # 概率保底：有效套餐优惠不能降低同一商品无优惠时的下单概率
    order_prob = max(base_order_prob, bundle_order_prob)
else:
    order_prob = base_order_prob
```

套餐优惠通过两重途径提升下单概率：
1. **price_score 重算传导**：折后单价降低 → bundle_price_score 上升 → bundle_score 上升 → order_prob 上升
2. **bundle_bonus 加成**：额外的小幅加成，模拟"优惠套餐"本身的心理锚定效果（上限 0.06）

同时增加概率保底规则：当 `bundle_applied=true` 时，最终 `order_prob = max(base_order_prob, bundle_order_prob)`，确保有效优惠不会降低同一商品在无优惠条件下的下单概率。

## 硬约束条件

| 约束 | 处理方式 |
|---|---|
| `purchase_limit` | 按买家维度限制套餐购买次数；同一买家达到上限后不再命中该套餐，其他买家不受影响 |
| 库存不够 N 件 | 先过滤出库存可支撑的阶梯，再逐级滚概率；若没有可用阶梯则退回 1 件 |
| 与单品折扣互斥 | 同一商品同时有单品折扣和套餐时，比较单品折扣价与 bundle 每件折后价，取每件价更低的一方 |
| 无变体商品 | 直接对 `listing.price` 计算 |
| 多变体商品 | 先选变体，再对该变体查询 bundle |

## purchase_limit 的处理

purchase_limit 不参与概率计算，也不作为活动级全局售罄开关。当前项目按“买家维度限购”处理：同一买家已生成的该套餐订单数达到 `purchase_limit` 后，本次不再命中该套餐；其他买家仍可继续购买。

说明：如果把 `purchase_limit=1` 误当成 campaign 累计上限，会导致真实活动只生成 1 单套餐后，后续买家即使买到组合 SKU 也全部退回普通单品随机数量；正确口径是每个买家最多购买指定次数。

## 数据加载：_load_ongoing_bundle_map

仿照现有 `_load_ongoing_discount_map`，按 `campaign_type == "bundle"` 加载：

```python
def _load_ongoing_bundle_map(
    db: Session, *, run_id: int, user_id: int, tick_time: datetime
) -> dict[int, dict[str, Any]]:
    campaigns = (
        db.query(ShopeeDiscountCampaign)
        .options(selectinload(ShopeeDiscountCampaign.items))
        .filter(
            ShopeeDiscountCampaign.run_id == run_id,
            ShopeeDiscountCampaign.user_id == user_id,
            ShopeeDiscountCampaign.campaign_type == "bundle",
            ShopeeDiscountCampaign.campaign_status.in_(("ongoing", "upcoming")),
            ShopeeDiscountCampaign.start_at <= tick_time,
            ShopeeDiscountCampaign.end_at >= tick_time,
        )
        .all()
    )

    bundle_map = {}
    for campaign in campaigns:
        rules = campaign.rules_json  # 含 bundle_type, purchase_limit, tiers
        purchase_limit = rules.get("purchase_limit")

        for item in campaign.items:
            bundle_map[int(item.listing_id)] = {
                "campaign_id": campaign.id,
                "bundle_type": rules["bundle_type"],
                "purchase_limit": purchase_limit,
                "tiers": rules["tiers"],
            }
    return bundle_map
```

## 验收标准

1. 创建一个 bundle 活动后，订单模拟日志中能看到 `bundle_applied: true` 和 `bundle_qty > 1` 的记录
2. `effective_unit_price` 按三种 bundle_type 正确计算
3. bundle 命中后会用 `selected_unit_price` 重算 `price_score`、`bundle_score` 和 `bundle_order_prob`
4. bundle 命中后的最终 `order_prob` 不低于同一商品无优惠时的 `base_order_prob`
5. `purchase_limit` 按买家维度生效：同一买家达到上限后不再命中该 bundle，不同买家仍可继续命中
6. 与单品折扣互斥：同一商品不会同时应用两种折扣，并选择每件价更低的一方
7. 库存不足时，只在库存可支撑的阶梯中逐级判断；无可用阶梯时退回 1 件
8. 买家决策流水（debug_payload）中记录 bundle 阶梯选择过程

# 52-Shopee 售前商品细节追问前后端数据库 Redis 实现设计

> 创建日期：2026-05-12  
> 状态：设计完成，尚未实现

## 1. 目标

在 Shopee 虚拟客服对话系统 MVP 中，优先实现“商品细节追问”售前客服案例。该案例在商品上架后，基于商品信息完整度、质量评分、规格复杂度和触发概率生成买家咨询会话，由 LLM 扮演买家向学生追问商品细节，学生在现有 Chat 抽屉或客服网页版中回复。

本设计只覆盖售前商品细节追问的后端接口、数据库、Redis、触发规则和前端数据接入口径。前端样式和布局沿用当前已完成页面，不调整视觉、间距、宽度或组件结构。

## 2. 范围与非范围

### 2.1 本期范围

- 商品上架后自动生成售前商品细节追问候选。
- 按游戏时间延迟、概率、质量评分、规格复杂度和去重规则创建客服会话。
- 会话绑定 `run_id/user_id/listing_id/buyer_profile_id/scenario_code`。
- 买家首条消息由 LLM 生成，围绕商品标题、规格、材质、颜色、尺码、适用场景、库存或发货信息追问。
- 学生发送消息后，后端写入学生消息并调用 LLM 生成买家下一条消息。
- 学生主动结束会话后，后端生成满意度评分并落库。
- Chat 抽屉和 `/shopee/customer-service/web` 共用同一套接口和消息数据。
- 自动回复作为可选辅助消息，不替代学生人工回复。
- 快捷回复仅作为输入辅助预留，本期不实现输入框推荐逻辑。
- 全部触发、延迟、频控、消息时间和评分时间使用对局游戏时间。

### 2.2 非范围

- 不修改前端样式和布局。
- 不实现物流停滞催单和签收破损退款。
- 不做教师手动派发客服案例。
- 不接入真实 Shopee Chat API。
- 不让 LLM 修改商品、订单、库存、发货或退款状态。
- 不重做自动回复和快捷回复配置页。
- 不实现复杂剧本后台编辑器，商品细节追问剧本由后端 seed 固化。
- 不立即把满意度接入订单概率；首版先落库和展示。

## 3. 现有系统承接点

| 现有模块 | 承接方式 |
| --- | --- |
| 商品发布/我的产品 | 商品上架后进入售前咨询触发候选池 |
| 商品质量评分 | 质量分越低，商品细节追问触发概率越高 |
| 商品规格/描述/图片 | 作为 LLM 买家追问和上下文展示来源 |
| 买家池/订单模拟 | 复用现有买家画像生成买家名称、语气和耐心程度 |
| Chat 抽屉 | 展示未处理售前咨询，支持查看与回复 |
| 客服网页版 | 展示完整会话列表、消息流和商品上下文 |
| 自动回复 | 买家进线后可写入辅助回复消息 |
| 快捷回复 | 后续在学生输入框中辅助插入常用回复 |

## 4. 核心流程

```text
商品发布成功 / 商品状态变为上架
      ↓
写入或检查商品细节追问候选
      ↓
按游戏时间延迟进入可触发窗口
      ↓
检查 run 状态、未处理会话数、同商品去重、每日上限、概率
      ↓
创建客服会话 shopee_customer_service_conversations
      ↓
LLM 基于商品信息和买家画像生成买家首条消息
      ↓
学生在 Chat 抽屉或客服网页版回复
      ↓
后端写入学生消息并生成买家下一条回复
      ↓
学生主动结束会话
      ↓
后端评分并写入满意度
```

## 5. 触发规则

### 5.1 触发入口

商品细节追问属于事件型触发，建议在以下位置接入：

- 商品创建并成功上架后。
- 商品从草稿、下架、审核中变为可售状态后。
- 商品质量评分完成或刷新后，如果商品已上架且尚未触发过本案例，可重新进入候选判断。

### 5.2 游戏时间延迟

商品上架后不立即创建会话，而是延迟一段游戏时间进入候选池：

| 项 | 建议值 | 说明 |
| --- | --- | --- |
| 最短延迟 | 1 游戏小时 | 避免刚上架立即进线 |
| 最长延迟 | 6 游戏小时 | 模拟买家浏览后咨询 |
| 候选有效期 | 2 游戏日 | 超过后未触发则跳过 |

延迟和有效期均以对局游戏时间计算，不使用服务器真实时间。

### 5.3 概率计算

基础概率来自剧本配置 `base_probability`，建议 MVP 默认 `0.35`。在基础概率上叠加商品质量与复杂度修正：

| 条件 | 修正建议 |
| --- | --- |
| 商品质量分 `< 60` | `+0.25` |
| 商品质量分 `60-79` | `+0.10` |
| 商品描述字数较短 | `+0.10` |
| 规格数量 `>= 3` | `+0.10` |
| 主图数量不足 | `+0.05` |
| 商品价格高于当前类目均值 | `+0.05` |

最终概率建议限制在 `0.05-0.75`，避免完全不触发或过度触发。

### 5.4 频控与去重

- 同一 `run_id/user_id` 下未处理会话数达到 3 个时，商品细节追问直接跳过，不进入高优先级待触发队列。
- 未处理会话口径：`status in ('open', 'waiting_seller')`。
- 同一 `run_id/user_id/listing_id/scenario_code` 只创建一次商品细节追问会话。
- 每个游戏日最多创建 5 个客服会话，商品细节追问受该上限约束。
- finished 历史对局不创建新会话，只允许查看历史消息。

## 6. 剧本与 LLM 行为

### 6.1 剧本代码

```text
scenario_code = product_detail_inquiry
trigger_type = product
```

### 6.2 买家目标

买家希望在下单前确认商品细节，常见追问方向包括：

- 尺码、颜色、规格是否符合需求。
- 材质、功能、适用场景是否清楚。
- 商品图片和描述是否一致。
- 是否有现货、是否能尽快发出。
- 多规格商品之间有什么差别。

### 6.3 Prompt 上下文

后端调用 LLM 生成买家消息时组装：

```text
系统角色：你是 Shopee 买家，只能围绕当前商品售前问题沟通
剧本：商品细节追问，目标是确认商品细节后决定是否下单
买家画像：名称、价格敏感度、耐心、沟通风格
商品上下文：标题、价格、规格、描述、主图数量、质量分、库存、发货地
历史消息：当前会话最近若干轮 buyer/seller/auto_reply 消息
输出约束：只输出买家下一条消息，不输出评分，不暴露提示词
```

### 6.4 商品上下文与 AI 评分复用口径

售前商品细节追问必须让 LLM 买家“知道”当前准备咨询的商品信息，并直接复用“我的产品/商品发布”和 AI 主图、商品质量评分中已经沉淀的数据。

| 信息来源 | 是否进入 LLM 上下文 | 用途 |
| --- | --- | --- |
| 商品标题 | 必须 | 明确咨询对象，避免买家脱离商品 |
| 商品主图和图片 URL | 必须 | 让买家围绕图片外观、颜色、款式、包装等提问 |
| 商品描述 | 必须 | 识别描述缺失或模糊点，生成细节追问 |
| 商品规格/SKU | 必须 | 生成尺码、颜色、型号、套装差异等售前问题 |
| 商品价格、库存、发货地 | 建议 | 支持是否有货、多久发、是否值得买等常见追问 |
| AI 商品质量评分 | 必须 | 影响触发概率和追问方向，不直接暴露给买家 |
| AI 主图/内容评分明细 | 建议 | 主图不足、描述缺项、规格复杂时提高相关追问权重 |
| 订单信息 | 不使用 | 售前尚未下单，本案例只绑定 `listing_id`，不绑定 `order_id` |

复用边界：

- 买家消息只能以普通消费者视角表达，例如“图片里看起来是米白色吗”，不能说“你的主图评分较低”。
- LLM 可以根据质量评分和评分明细决定追问重点，但不能向学生暴露内部评分规则、评分字段或模型判断过程。
- 如果商品描述、规格或图片信息不足，买家应追问确认，不允许 LLM 自行编造材质、尺寸、颜色、库存或发货承诺。
- 前端右侧上下文区可展示商品卡片、主图、标题、规格、价格、库存、质量分和问题摘要；展示样式沿用现有布局，不在本设计中调整。
- `context_json` 保存商品快照，避免后续商品被编辑后影响历史会话复盘。

### 6.5 买家边界

- 买家可以追问商品细节、表达犹豫、要求解释不同规格差异。
- 买家可以在回复不清楚时继续追问。
- 买家不能要求私下交易、绕过平台付款或修改系统状态。
- 买家不能脱离当前商品闲聊。
- 买家不能透露评分标准。

### 6.6 对话轮次与结束条件

售前商品细节追问不固定死对话轮数，采用“最少 5 条、推荐 7 条、最多 50 条消息”的教学闭环。

| 消息数 | 口径 | 说明 |
| --- | --- | --- |
| 最少 5 条 | 买家首问、学生回复、买家追问、学生补充、买家收口 | 保证学生至少经历一次追问与补充解释 |
| 推荐 7 条 | 在 5 条基础上增加一轮买家确认和学生最终引导 | 更适合作为标准售前训练节奏 |
| 最多 50 条 | 买家连续追问或学生答非所问时允许扩展 | 防止 LLM 无限追问拖慢经营节奏 |

推荐 7 条消息的标准流程：

| 序号 | 角色 | 目的 |
| --- | --- | --- |
| 1 | 买家 | 发起商品细节问题 |
| 2 | 学生 | 回答核心问题 |
| 3 | 买家 | 追问规格、材质、颜色、尺码或适用场景 |
| 4 | 学生 | 补充解释并减少误购风险 |
| 5 | 买家 | 表达犹豫或确认关键点 |
| 6 | 学生 | 给出最终购买建议或平台内下单引导 |
| 7 | 买家 | 收口，表示感谢、决定下单或再考虑 |

结束规则：

- 少于 5 条消息时允许学生点击结束，但评分应识别为可能过早结束。
- 达到 7 条消息且买家目标已满足时，LLM 应进入收口状态，不继续制造新问题。
- 达到 50 条消息后，系统应阻止继续自动生成买家追问，并提示学生结束会话或做最后补充。
- 如果学生答非所问，买家可以继续追问，但仍受 50 条消息上限约束。
- 会话结束后进入满意度评分，不再生成新的买家消息。

## 7. 满意度评分

学生主动结束会话后，后端基于会话消息和商品上下文生成评分。

| 维度 | 分值 | 商品细节追问评分重点 |
| --- | --- | --- |
| 响应完整度 | 0-30 | 是否回答买家核心问题 |
| 商品准确性 | 0-25 | 是否基于商品真实标题、规格、描述和库存回答 |
| 服务态度 | 0-20 | 是否礼貌、耐心、没有催促或敷衍 |
| 购买引导 | 0-15 | 是否给出合理购买建议，减少误购 |
| 平台合规 | 0-10 | 是否未引导私下交易或承诺虚假信息 |

满意度等级：

| 等级 | 分数区间 | 处理 |
| --- | --- | --- |
| `high` | 85-100 | 记录高满意，可后续作为复购倾向轻微加成 |
| `medium` | 60-84 | 记录正常完成 |
| `low` | 0-59 | 记录低满意，可后续作为转化降低或差评风险参考 |

首版只落库和展示评分，不立即影响订单模拟概率。

## 8. 数据库设计

本期可复用 51 号总设计中的客服通用表，并针对商品细节追问 seed 一条剧本数据。

### 8.1 `shopee_customer_service_scenarios`

新增或 seed：

| 字段 | 值 |
| --- | --- |
| `scenario_code` | `product_detail_inquiry` |
| `name` | `商品细节追问` |
| `trigger_type` | `product` |
| `enabled` | `true` |
| `base_probability` | `0.35` |
| `cooldown_game_hours` | `48` |
| `buyer_persona_prompt` | 售前买家画像提示词 |
| `scenario_prompt` | 商品细节追问目标和边界 |
| `rubric_json` | 本文第 7 节评分规则 |

### 8.2 `shopee_customer_service_conversations`

商品细节追问会话要求：

- `scenario_code='product_detail_inquiry'`。
- `listing_id` 必填。
- `order_id` 为空。
- `context_json` 保存商品快照，至少包含标题、价格、规格、描述摘要、主图数量、主图/图片 URL、质量分、质量评分问题摘要、AI 主图/内容评分明细、库存、发货地。
- `trigger_reason` 保存触发原因，例如：`商品质量分较低且规格较多，买家发起售前咨询`。
- 建议增加唯一约束或代码级去重：`run_id/user_id/scenario_code/listing_id`。

### 8.3 `shopee_customer_service_messages`

消息来源：

| `sender_type` | 说明 |
| --- | --- |
| `buyer` | LLM 生成的买家消息 |
| `seller` | 学生发送的消息 |
| `auto_reply` | 自动回复辅助消息 |
| `system` | 系统提示或会话状态消息 |

`sent_game_at` 使用对局游戏时间。

### 8.4 可选候选表

如现有触发链路不适合直接扫描商品，可新增候选表：`shopee_customer_service_trigger_candidates`。

| 字段 | 类型 | 注释 |
| --- | --- | --- |
| `id` | Integer | 候选 ID |
| `run_id` | Integer | 所属对局 ID |
| `user_id` | Integer | 所属用户 ID |
| `scenario_code` | String(64) | 剧本代码 |
| `biz_type` | String(32) | 业务对象类型，商品为 `listing` |
| `biz_id` | Integer | 商品 ID |
| `status` | String(32) | pending/triggered/skipped/expired |
| `eligible_game_at` | DateTime | 可触发游戏时间 |
| `expire_game_at` | DateTime | 过期游戏时间 |
| `reason_json` | JSON | 候选原因与概率修正明细 |
| `created_at` | DateTime | 创建时间 |
| `updated_at` | DateTime | 更新时间 |

MVP 如果能通过商品表和会话表直接判断，也可以暂不新增候选表，减少实现复杂度。

## 9. 后端接口设计

接口沿用客服通用路径，优先满足当前已完成前端页面的数据接入。

### 9.1 会话列表

```http
GET /shopee/runs/{run_id}/customer-service/conversations?scenario=product_detail_inquiry&status=open&page=1&page_size=20
```

返回字段建议：

```json
{
  "items": [
    {
      "id": 1,
      "scenario_code": "product_detail_inquiry",
      "scenario_name": "商品细节追问",
      "buyer_name": "Aisyah",
      "status": "open",
      "last_message": "Hi, may I know the material of this item?",
      "last_message_game_at": "2026-05-12 10:00:00",
      "listing": {
        "id": 12,
        "title": "...",
        "image_url": "..."
      },
      "unread_count": 1
    }
  ],
  "total": 1,
  "page": 1,
  "page_size": 20
}
```

### 9.2 会话详情

```http
GET /shopee/runs/{run_id}/customer-service/conversations/{conversation_id}
```

返回：

- 会话基础信息。
- 商品上下文卡片数据。
- 消息列表。
- 满意度评分状态。
- 是否允许发送消息和结束会话。

### 9.3 发送学生消息

```http
POST /shopee/runs/{run_id}/customer-service/conversations/{conversation_id}/messages
```

请求：

```json
{
  "content": "您好，这款有两个颜色，材质是棉混纺，日常通勤比较适合。"
}
```

后端行为：

1. 校验 run 未 finished。
2. 校验会话属于当前 `run_id/user_id` 且状态允许回复。
3. 写入 `seller` 消息。
4. 清理会话详情和列表缓存。
5. 调用 LLM 生成 `buyer` 下一条消息。
6. 写入买家消息。
7. 返回新增消息或完整消息列表。

### 9.4 结束会话并评分

```http
POST /shopee/runs/{run_id}/customer-service/conversations/{conversation_id}/resolve
```

后端行为：

1. 校验 run 未 finished。
2. 将会话状态置为 `resolved`。
3. 调用评分 prompt 或规则模型生成评分。
4. 写入 `satisfaction_score/satisfaction_level/score_detail_json/closed_game_at`。
5. 清理会话详情、列表和统计缓存。
6. 返回评分结果。

### 9.5 触发器接口或服务

不建议暴露给前端。后端内部服务建议提供：

```python
trigger_product_detail_inquiry(run_id, user_id, listing_id, game_now)
scan_product_detail_inquiry_candidates(run_id, user_id, game_now)
```

调用来源可以是商品上架成功后的同步触发，也可以是对局 tick/订单模拟 tick 中的巡检服务。

## 10. Redis 设计

| Key | 用途 | TTL |
| --- | --- | --- |
| `shopee:customer_service:conversations:{run_id}:{user_id}:{hash}` | 会话列表缓存 | 30s |
| `shopee:customer_service:conversation:{conversation_id}` | 会话详情缓存 | 30s |
| `shopee:customer_service:trigger_lock:{run_id}:{user_id}:product_detail_inquiry:{listing_id}` | 同商品触发锁 | 48 游戏小时对应真实 TTL 或持久去重兜底 |
| `shopee:customer_service:llm_rate:{run_id}:{user_id}` | LLM 调用限流 | 60s |
| `shopee:customer_service:model_settings:{run_id}:{user_id}` | 客服模型配置缓存 | 300s |

缓存失效：

- 创建会话后清理会话列表缓存。
- 发送消息后清理会话详情和列表缓存。
- 结束会话评分后清理会话详情、列表和统计缓存。
- 更新模型配置后清理模型配置缓存。

Redis 只做缓存、限流和短期锁，最终去重以数据库会话记录为准。

## 11. 前端接入口径

本期不修改前端样式和布局，只把已有页面接入接口数据。

### 11.1 Chat 抽屉

- 加载未处理会话列表。
- 显示买家名称、最新消息、关联商品标题或缩略图。
- 点击会话打开消息流。
- 发送消息调用通用发送接口。
- “网页版”入口保持现有跳转。

### 11.2 客服网页版

- 左侧会话列表读取同一会话列表接口。
- 中间消息流读取会话详情接口。
- 输入框提交发送消息接口。
- 结束会话按钮调用 resolve 接口。
- 右侧上下文卡片展示商品快照和评分结果。

### 11.3 历史对局

finished 对局：

- 可以查看会话列表、详情和评分。
- 不允许发送消息。
- 不允许结束未结束会话。
- 不触发新的商品细节追问会话。

## 12. 自动回复与快捷回复关系

- 如果默认自动回复启用，买家首条消息写入后，可以追加一条 `sender_type=auto_reply` 的辅助回复。
- 自动回复内容不直接完成评分，学生仍需人工处理核心问题。
- 快捷回复本期只读取既有配置作为后续预留，不实现关键词推荐弹窗。
- 使用快捷回复发送的内容仍按 `seller` 消息评分。

## 13. 验收标准

### 13.1 触发与去重

- 商品上架后，达到游戏时间延迟且通过概率判断时创建商品细节追问会话。
- 同一商品同一对局不会重复创建商品细节追问会话。
- 未处理会话达到 3 个时，不再新增商品细节追问。
- finished 对局不会触发新会话。

### 13.2 会话与消息

- 创建会话后自动生成买家首条消息，内容围绕商品细节。
- Chat 抽屉和客服网页版能看到同一会话和同一消息。
- 学生发送消息后，数据库写入 `seller` 消息，并生成新的 `buyer` 回复。
- 任一入口发送后，另一入口刷新可见最新消息。

### 13.3 上下文与评分

- 会话详情展示商品标题、规格、描述摘要、价格、图片和质量分。
- 学生结束会话后生成满意度分数、等级和维度详情。
- 评分重点能识别是否准确回答商品细节、是否礼貌、是否减少误购。

### 13.4 模型配置与降级

- 客服模型未配置或禁用时，不调用 LLM，并返回明确配置提示。
- API Key 不明文返回前端。
- LLM 调用失败时，已写入的学生消息保留，会话状态不错误关闭。

## 14. 实现顺序建议

1. 新增客服通用 ORM、历史库保障、表注释和字段注释。
2. seed `product_detail_inquiry` 剧本。
3. 实现会话列表、详情、发送消息、结束评分接口。
4. 实现商品上架后的商品细节追问触发服务。
5. 接入 Redis 列表/详情缓存、触发锁和 LLM 限流。
6. 在不改前端样式布局的前提下，把 Chat 抽屉和客服网页版接入接口。
7. 做窄验证：商品上架触发、发送消息、结束评分、finished 只读、双入口一致。

## 15. 风险与约束

- 触发过多会打扰学生，应严格遵守 3 个未处理会话上限和每日上限。
- 商品信息不足时，LLM 容易编造细节，prompt 必须要求只基于上下文回答，未知信息应追问或表达不确定。
- Redis 锁不能作为唯一去重依据，数据库会话记录必须兜底。
- 前端已完成样式布局，接口接入时只替换数据源和事件处理，不做视觉重排。

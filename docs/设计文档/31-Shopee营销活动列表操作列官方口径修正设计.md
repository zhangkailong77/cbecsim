# 31-Shopee 营销活动列表操作列官方口径修正设计

## 1. 目标

根据 Shopee 官方页面截图，修正营销中心折扣页中不同活动 Tab 的操作列展示口径，避免套餐优惠、加价购/满额赠沿用单品折扣的“数据/查看数据”等不符合官方页面的操作项。

本设计仅整理待修改口径，不立即改代码。

## 2. 背景与问题

当前折扣活动列表已经支持多类活动展示：

- 单品折扣（Discount Promotions）
- 套餐优惠（Bundle Deal）
- 加价购/满额赠（Add-on Deal / Gift with Min. Spend）

此前为复用单品折扣详情页与数据页能力，加价购/满额赠活动列表操作列也接入了“详情 / 数据”等入口。但对照官方截图后发现，不同 Tab 下的操作项不一致：

- 套餐优惠 Tab 的操作列是 `Edit / Duplicate / Details / Orders`；
- 加价购 Tab 的操作列是 `Details / Duplicate / Orders`；
- 加价购/满额赠 Tab 不展示 `Edit`，也不展示 `Data`；
- 套餐优惠 Tab 不展示 `Data`，而是展示 `Orders`。

因此需要按 Tab 与活动类型区分操作列文案和点击行为。

## 4. 修正范围

### 4.1 活动列表 Tab 口径

| Tab | 覆盖活动 | 操作列中文显示 |
| --- | --- | --- |
| All | 全部活动 | 按活动类型决定操作列，不强行统一 |
| Discount Promotions | 单品折扣 | 保持现有单品折扣操作口径 |
| Bundle Deal | 套餐优惠 | 编辑 / 复制 / 详情 / 订单 |
| Add-on Deal | 加价购、满额赠 | 详情 / 复制 / 订单 |

### 4.2 活动类型映射

| 后端类型 | 真实活动类型 | 所属 Tab | 操作列口径 |
| --- | --- | --- | --- |
| `discount` | 单品折扣 | Discount Promotions | 单品折扣原有口径 |
| `bundle` | 套餐优惠 | Bundle Deal | 编辑 / 复制 / 详情 / 订单 |
| `add_on` + `promotion_type=add_on` | 加价购 | Add-on Deal | 详情 / 复制 / 订单 |
| `add_on` + `promotion_type=gift` | 满额赠 | Add-on Deal | 详情 / 复制 / 订单 |

## 5. 页面行为设计

### 5.1 详情

- 套餐优惠：进入套餐优惠详情页或现有兼容详情页。
- 加价购/满额赠：进入加价购/满额赠详情页。
- 中文显示统一为“详情”。

### 5.2 复制

- 中文显示为“复制”。
- 后续点击行为建议：进入对应创建页，并携带源活动 ID 作为复制来源。
- 若当前未实现复制逻辑，可先展示入口但暂时禁用，或点击后给出“暂未开放”的轻提示。

### 5.3 编辑

- 仅套餐优惠 Tab 展示“编辑”。
- 加价购/满额赠 Tab 不展示“编辑”。
- 若套餐优惠活动已结束或只读历史对局，编辑入口应按现有只读规则禁用或隐藏。

### 5.4 订单

- 中文显示为“订单”。
- 套餐优惠和加价购/满额赠均展示。
- 后续点击行为建议优先复用已有订单页，并通过 query 参数携带营销活动筛选条件：
  - `marketing_campaign_type`
  - `marketing_campaign_id`
  - 必要时携带 `source_campaign_id` 或正式活动 ID。

### 5.5 数据页入口

- 套餐优惠 Tab 不展示“数据 / 查看数据”。
- 加价购/满额赠 Tab 不展示“数据 / 查看数据”。
- 已实现的数据页接口和页面可以保留为内部能力，但不作为这两个官方 Tab 的列表操作入口。

## 6. 前端实现建议

### 6.1 操作列生成规则

在活动列表渲染时，不再使用一套统一 actions，而是按当前 Tab 与活动类型生成：

```ts
function getPromotionActions(row, activeTab) {
  if (activeTab === 'bundle' || row.campaign_type === 'bundle') {
    return ['编辑', '复制', '详情', '订单'];
  }
  if (activeTab === 'add_on' || row.campaign_type === 'add_on') {
    return ['详情', '复制', '订单'];
  }
  return getDiscountPromotionActions(row);
}
```

实际实现时需沿用现有类型命名和 Tab key，不强制使用上述伪代码命名。

### 6.2 All Tab 处理

All Tab 中可能混合多种活动，建议按每一行的真实活动类型决定操作列：

- `discount` 行：使用单品折扣操作；
- `bundle` 行：使用套餐优惠操作；
- `add_on/gift` 行：使用加价购操作。

这样可避免 All Tab 下出现与活动类型不匹配的操作项。

## 7. 后端影响

本次修正优先属于前端展示与路由行为调整，后端不一定需要新增接口。

可能涉及的后端支持：

- 若“订单”入口需要精准筛选活动订单，订单列表接口需支持营销活动筛选参数；
- 若“复制”入口需要预填活动配置，需复用或补充已有 bootstrap/source campaign 读取能力；
- 若仅先修正操作列文案和隐藏“数据”入口，可不改后端。

## 8. 验收标准

1. Bundle Deal Tab 活动列表操作列显示：`编辑 / 复制 / 详情 / 订单`。
2. Add-on Deal Tab 活动列表操作列显示：`详情 / 复制 / 订单`。
3. Bundle Deal Tab 不再显示“数据 / 查看数据”。
4. Add-on Deal Tab 不再显示“数据 / 查看数据”。
5. Add-on Deal Tab 不显示“编辑”。
6. All Tab 中不同活动按各自类型显示对应操作项。
7. 单品折扣 Tab 原有操作口径不被套餐优惠、加价购修正影响。
8. 历史回溯只读场景下，编辑/复制类操作遵守现有只读限制。

# ad02-已结束对局续期与追加游戏年限设计

## 1. 目标
- 支持 `super_admin` 对已结束（`finished`）对局执行“续期恢复”，让该局在不重开新局的前提下继续经营，便于调试、教学和运营演示。
- 续期时保持 `created_at`（开始时间）不变，仅延长结束时间，并让游戏总年限同步追加，而不是把原来的 1 年时间轴重新拉伸。
- 保证“7天 = 1个经营年”的时间比例在续期后继续成立：如果一个 `7天 -> 365游戏日` 的对局续期 `+7天`，则总时间轴变为 `14天 -> 730游戏日`。
- 让历史回溯、只读闸门、对局结束判定、管理员干预入口之间的语义保持一致，避免“结束后被重开但仍按历史局展示”的状态冲突。

## 2. 场景说明
### 2.1 适用场景
- 一场 7 天对局已经自然结束，但管理员希望继续用这局数据做联调或演示，不想要求玩家重新开一局。
- 教学/培训场景中，需要把已经结束的局恢复为可经营状态，并继续追加新的经营年限。
- 某局在结束前后沉淀了大量有效经营数据，管理员希望基于这批历史数据继续推进后续链路验证。

### 2.2 非目标
- 本期不做“复制一局生成调试沙盒”的旁路方案，仍然是在原 `run_id` 上续期。
- 本期不重做所有业务模块的数据模型，只围绕 `GameRun` 的时间轴与管理员续期能力扩展。
- 本期不支持同一玩家同时存在多个 `running` 对局；若已存在新的运行中对局，不允许把旧 `finished` 局直接恢复成第二个 `running`。
- 本期不做复杂审批流，仅沿用 `super_admin` 直达操作。

## 3. 当前实现现状
### 3.1 已有基础
- `game_runs` 已包含 `duration_days / day_index / status / created_at` 字段，可表达对局时长、状态和起始时间。
- 当前结束时间按 `created_at + duration_days` 计算。
- 当前管理员已经有“对局管理”页面与运行中对局延长能力（ad01 方案），但只覆盖 `running` 对局。
- 历史经营记录页只展示 `finished` 对局，并以只读模式回溯。

### 3.2 当前限制
- 管理员运行中对局列表仅返回 `running` 对局。
- 当前延长接口会拒绝 `finished` 对局，不支持“结束后恢复”。
- 前后端当前都把总游戏天数写死为 `365`，即：
  - 后端游戏日映射固定使用 `TOTAL_GAME_DAYS = 365`
  - 前端倒计时/游戏日换算也固定使用 `365`
- 因此当前即使把真实时长从 `7天` 改到 `14天`，也只会把同一年的经营节奏拉慢，不会变成“再追加一年”。

### 3.3 当前历史局语义
- `history/options` 只返回 `finished` 对局。
- `history/summary` 仅支持 `finished` 对局查看。
- 前端 `/setup/history` 页面明确把 `finished` 视为历史只读局。
- 如果未来允许把 `finished` 局恢复为 `running`，则该局应立即退出“历史只读”集合，重新回到可经营集合。

## 4. 核心问题判断
### 4.1 仅修改 `duration_days` 不够
如果只把已结束对局的 `duration_days` 变大，会有两个核心问题：
1. **状态问题**：`finished` 对局仍然是历史只读语义，不能自动恢复经营能力。
2. **时间问题**：当前总游戏天数固定 365，只改真实时长只会“拉伸同一年”，不会“追加新的一年”。

### 4.2 方案 B 的本质
方案 B 不是“对结束局做一次 ad01 式延长”，而是引入一条新的管理员能力：
- 对 `finished` 对局进行“续期恢复”；
- 在延长真实结束时间的同时，追加该局的**总游戏天数**；
- 让该局从历史只读状态重新回到 `running`，并继续经营。

## 5. 核心设计原则
### 5.1 开始时间固定，不回写历史经营数据
- `created_at` 永远不变。
- 续期不会重置资金、订单、物流、库存、财务、营销等经营数据。
- 续期只改“时间轴”和“状态”，不篡改经营结果。

### 5.2 真实时间与游戏时间同时扩容
- 真实结束时间向后延长。
- 游戏总时长也同步增加，不再固定为 `365`。
- 已经走过的游戏日不回退、不压缩，而是继续向后累计。

### 5.3 单玩家仍只允许一个 `running` 对局
- 当前系统大量接口默认围绕“当前玩家的唯一运行中对局”组织。
- 为了避免引入系统级歧义，续期恢复时必须继续保持这个约束。
- 因此：若玩家当前已有另一场 `running` 对局，则旧 `finished` 局默认不允许恢复。

### 5.4 历史/当前态必须单向切换清晰
- `finished` -> 进入历史只读集合。
- 管理员续期恢复后：`finished -> running`。
- 恢复成功后，该局应从历史列表移除，并重新进入可经营集合。

## 6. 时间规则设计
### 6.1 现有规则
当前系统隐含规则是：
- `duration_days` 表示真实世界运行天数；
- 总游戏天数固定 `365`；
- 游戏日换算公式本质为：

`当前游戏进度 = 已经过的真实时间 / 总真实时长 * 365`

### 6.2 方案 B 的目标规则
方案 B 下，每个对局需要有自己的**总游戏天数**，而不再共享全局固定 `365`。

推荐引入以下概念：
- `effective_end_time`：对局当前有效结束时间；
- `total_game_days`：该局当前总游戏天数；
- `base_real_duration_days`：该局初始真实时长快照；
- `base_game_days`：该局初始游戏天数快照，默认 `365`。

### 6.3 追加规则
当管理员把已结束对局继续延长时：
- `effective_end_time` 变大；
- `total_game_days` 也按原始时间比例同步增加。

推荐公式：

`追加游戏天数 = round(追加真实时长 / 初始真实时长 * 初始游戏天数)`

例如：
- 初始规则：`7天 -> 365游戏日`
- 管理员续期 `+7天`
- 则：`追加游戏天数 = 7 / 7 * 365 = 365`
- 最终：`14天 -> 730游戏日`

### 6.4 为什么要锁定“初始比例”
不建议每次续期都按“当前总时长 / 当前总游戏天数”反推比例，否则多次续期后可能产生累计舍入漂移。

推荐做法：
- 在对局创建时锁定一组基础比例：
  - `base_real_duration_days`
  - `base_game_days`
- 后续所有续期都按这组基础比例追加游戏天数。

### 6.5 游戏日换算新公式
后续全链路统一改为：

`当前游戏进度 = 已经过的真实时间 / 总真实时长 * total_game_days`

而不是固定除以/乘以 `365`。

这样在“7天续到14天”的场景中：
- 第 7 天整时，进度会自然落到 `Day 366` 左右；
- 不是回到更低的游戏日，也不是把 365 天重新摊薄。

## 7. 管理员操作定义
### 7.1 操作名称建议
建议在管理员侧新增一类动作：
- `续期恢复`

而不是仍然叫“延长对局”，因为它不仅是延长结束时间，还包含：
- 结束局恢复为 `running`
- 追加游戏年限
- 历史只读态重新开放经营

### 7.2 支持的两类输入方式
建议支持两类管理员输入：

1. **按天续期**
- `+7天`
- `+14天`
- 自定义 `+N天`

2. **直接设置新的结束时间**
- 开始时间不变
- 管理员指定新的 `effective_end_time`
- 系统自动换算追加的真实时长与追加的游戏天数

### 7.3 建议的约束规则
- 新结束时间必须大于旧结束时间。
- 新结束时间必须晚于当前时间。
- 仅 `super_admin` 可操作。
- 仅允许对 `finished` 对局执行“续期恢复”；
  - `running` 对局继续沿用 ad01 的延长逻辑；
  - `abandoned` 仍不允许恢复。
- 若该玩家当前已有其他 `running` 对局，则本次恢复拒绝。

## 8. 数据模型建议
### 8.1 `game_runs` 建议新增字段
推荐新增以下字段：

- `base_real_duration_days`：初始真实时长快照（创建时写入）
- `base_game_days`：初始游戏天数快照，默认 `365`
- `total_game_days`：当前总游戏天数，默认 `365`
- `manual_end_time`：管理员手动指定的结束时间（可为空）

### 8.2 保留字段用途
- `duration_days`：保留，继续作为“整天级时长展示/兼容字段”；
- `created_at`：不变，始终作为时间轴起点；
- `status`：恢复时从 `finished` 改回 `running`；
- `day_index`：继续作为当前游戏日快照字段，但其计算来源应改为动态 `total_game_days`。

### 8.3 实际结束时间来源建议
推荐统一定义：
- 若 `manual_end_time` 存在，则以其作为 `effective_end_time`
- 否则仍回退为 `created_at + duration_days`

这样可以兼容：
- 旧数据（没有 `manual_end_time`）
- 新的管理员精确调时间场景

### 8.4 管理员操作日志表（建议新增）
建议新增专门日志表，例如：`game_run_admin_time_adjustments`

建议记录：
- `run_id`
- `user_id`
- `operator_user_id`
- `action_type`（如 `extend_running` / `renew_finished` / `set_end_time`）
- `old_status`
- `new_status`
- `old_end_time`
- `new_end_time`
- `old_total_game_days`
- `new_total_game_days`
- `reason`
- `created_at`

原因：
- 续期恢复属于强运营干预动作，后续排查时间轴问题时需要审计依据。

## 9. 后端方案
### 9.1 接口建议
建议在管理员接口下新增 finished 续期接口，例如：
- `POST /game/admin/runs/{run_id}/renew`

请求体建议：
```json
{
  "mode": "extend_days",
  "extend_days": 7,
  "reason": "调试继续推进 Step05"
}
```

或：
```json
{
  "mode": "set_end_time",
  "target_end_time": "2026-04-30T23:59:59Z",
  "reason": "教学演示追加一周"
}
```

返回建议包含：
- `run_id`
- `old_status`
- `new_status`
- `old_end_time`
- `new_end_time`
- `old_total_game_days`
- `new_total_game_days`
- `current_day_index`
- `remaining_seconds`

### 9.2 后端执行步骤建议
1. 校验管理员权限（`super_admin`）。
2. 查询目标 `run_id`。
3. 校验目标对局当前状态为 `finished`。
4. 校验该玩家当前不存在其他 `running` 对局。
5. 计算旧结束时间、旧总游戏天数。
6. 根据输入模式算出新的结束时间。
7. 按基础比例计算应追加的游戏天数。
8. 更新：
   - `status = running`
   - `manual_end_time` 或兼容更新后的时长字段
   - `total_game_days`
9. 清理相关缓存。
10. 写入管理员操作日志。
11. 返回最新状态。

### 9.3 结束判定兼容
后续结束判定应改为：
- 使用 `effective_end_time` 而不是仅 `created_at + duration_days`
- 运行中对局一旦达到新的 `effective_end_time`，再转回 `finished`

这样同一局可以经历：
- 第一次自然结束
- 管理员续期恢复
- 第二次自然结束

### 9.4 缓存清理建议
续期恢复后至少清理：
- 当前对局概览缓存
- 管理员买家池总览缓存
- 历史总结缓存
- 历史对局列表缓存
- 任何基于 `run.status` 或 `run_id` 的工作台摘要缓存

否则会出现：
- 后端已恢复为 `running`
- 历史页仍显示该局是已结束
- 工作台仍显示旧的结束时间或只读状态

## 10. 前端方案
### 10.1 管理页结构建议
建议在现有 `对局管理` 页中增加第二块视图：
- `运行中对局`
- `已结束对局（可续期恢复）`

其中：
- `running` 对局继续使用 ad01 的“延长”操作；
- `finished` 对局使用本次 ad02 的“续期恢复”操作。

### 10.2 页面展示字段建议
对于已结束对局，建议展示：
- 对局 ID
- 玩家
- 市场
- 当前状态（finished）
- 原始开始时间
- 当前结束时间
- 当前总游戏天数
- 最终停留游戏日
- 恢复后新的结束时间预览
- 恢复后新的总游戏天数预览

### 10.3 操作交互建议
点击“续期恢复”时，建议显式展示预览信息：
- 原结束时间
- 新结束时间
- 原总游戏天数
- 新总游戏天数
- 当前会从 `finished` 变回 `running`
- 若该玩家已有其他运行中对局，则直接提示不可恢复

### 10.4 历史页联动
恢复成功后：
- 该局应从 `/setup/history` 的列表中消失；
- 再次通过 `run_id` 进入阶段页时，不应再显示历史只读提示；
- 工作台右侧统一时间中心应按“当前 running 对局”口径重新展示。

### 10.5 时间中心与阶段页改造
前端所有依赖固定 `365` 的地方，都应改为读取 run 级别的 `total_game_days`。

重点包括：
- 总倒计时
- 游戏日映射
- 游戏时钟
- 剩余游戏天数
- 历史总结页展示
- 管理员选中对局后的右侧统一时间中心

## 11. 关键冲突与推荐处理
### 11.1 已结束旧局 vs 玩家已有新 running 局
这是方案 B 最大的业务冲突点。

推荐规则：
- **默认拒绝恢复**。
- 如果该玩家名下已经有新的 `running` 对局，则管理员不能把旧 `finished` 局恢复为第二个 `running`。

原因：
- 当前系统默认“一名玩家只有一个当前运行中对局”。
- 若放开双 running，会影响大量 `/runs/current`、工作台入口、阶段页默认加载、缓存 key 和经营写接口路由假设。

### 11.2 历史局恢复后的历史定位
推荐语义：
- 历史页只看**当前状态仍是 `finished`** 的局。
- 一旦恢复，该局不再属于历史集合。
- 第二次结束后，再重新进入历史集合。

### 11.3 手动设置结束时间的精度问题
若支持精确到时分秒的结束时间：
- 必须统一以后端 UTC 计算；
- 前端只负责展示本地格式，不参与结束判定；
- 续期新增游戏天数按真实秒数计算，不按前端表面天数近似。

## 12. 风险点
### 12.1 时间模型改动面较大
方案 B 需要把“总游戏天数固定 365”的假设从后端和前端同时改掉，影响面比 ad01 明显更大。

### 12.2 历史只读语义会变成动态集合
某局今天是历史局，续期恢复后就不再是历史局，历史页与阶段页只读逻辑都必须依赖**当前状态**而不是“曾经结束过”。

### 12.3 自动演化恢复风险
恢复成功后，Shopee 自动模拟、自动取消、自动物流推进、自动回款补记等写逻辑会重新生效。

这本身是目标行为，但必须确认：
- 老局里未完成订单重新恢复推进是否符合预期；
- 不会因为历史缓存残留而出现重复推进或脏读。

### 12.4 数据回填风险
旧数据没有 `base_real_duration_days / base_game_days / total_game_days / manual_end_time` 时，需要在迁移或读取兜底中补默认值，否则历史局恢复时无法准确计算追加规则。

## 13. 推荐实施顺序
1. 明确方案 B 的业务语义：`finished` 局可恢复、开始时间不变、结束时间延长、总游戏天数追加。
2. 在数据模型中补齐：基础比例、总游戏天数、手动结束时间、管理员操作日志。
3. 后端实现新的 `renew` 接口，并接入“单玩家仅一个 running 对局”的校验。
4. 后端统一把结束判定与游戏日换算改为 run 级动态字段，不再写死 `365`。
5. 前端 `对局管理` 页补齐 `finished` 续期恢复区块。
6. 前端统一时间中心、工作台、阶段页、历史页改为读取 run 级 `total_game_days`。
7. 做缓存清理与只读态/恢复态联调验证。
8. 补充自动化测试与手工回归。

## 14. 验收标准
### 14.1 功能验收
- 管理员可以在对局管理页看到 `finished` 对局，并对其执行续期恢复。
- 恢复后：
  - `status` 从 `finished` 变为 `running`
  - 开始时间不变
  - 结束时间变长
  - 总游戏天数同步增加
- 对于 `7天 -> 365游戏日` 的对局，续期 `+7天` 后，总游戏天数应变为 `730`。
- 在原结束时刻恢复后，该局应从 `Day 366` 左右继续推进，而不是重新摊薄到同一年的较低天数。
- 恢复成功后，该局从历史列表消失，并重新进入经营主链路。
- 第二次达到新结束时间后，该局可再次自然收口为 `finished`。

### 14.2 回归验收
- ad01 运行中对局延长逻辑不受影响。
- 现有历史经营记录页对仍为 `finished` 的局继续正常工作。
- 当前没有被续期恢复的局，行为完全不变。
- 若玩家已有新的 `running` 对局，旧 finished 局恢复应被明确拒绝。
- Shopee 订单域、物流、财务、阶段页在恢复后能继续工作，不出现“仍是历史只读”或“恢复后仍无法写入”的错乱状态。

## 15. 结论与建议
如果你的真实诉求是：
- 已结束对局还能继续调试；
- 开始时间不变，结束时间往后推；
- 不是把原本的一年拉慢，而是继续追加新的经营年限；

那么 ad02（方案 B）是更匹配的设计方向。

推荐结论：
- **不要**只在现有 ad01 上放开 finished 延长；
- **应该**把“续期恢复 finished 对局”作为一条独立管理员能力设计；
- **必须**把 `365` 固定总游戏天数改成 run 级动态字段，否则无法满足“7天再加1年”的语义；
- **默认保持单玩家仅一个 running 对局**，这是控制改动面和系统稳定性的关键边界。

这样设计后，系统可以同时支持：
- ad01：运行中对局延长真实时间
- ad02：已结束对局恢复运行，并追加新的游戏年限

# ad01-运行中对局延长设计方案

## 1. 目标
- 支持管理员对运行中的对局执行延长操作，例如 `+7天`、`+14天` 或自定义天数。
- 延长后，对局结束判定、工作台倒计时、历史经营记录入口、阶段页只读边界保持一致。
- 保证延长操作对玩家透明：已运行对局无需重开，后续经营可按新时长继续进行。
- 为后续扩展“后台可选 7/14/30 天”“统一时间规则配置”预留兼容空间。

## 2. 场景说明
### 2.1 适用场景
- 运营活动临时延长，避免对局过早结束。
- 教学/演示场景需要给某一批次玩家补充经营时间。
- 某局接近结束时，管理员希望人工续期而不是要求玩家重开一局。

### 2.2 非目标
- 本期不重做整套时间系统。
- 本期不改历史已结束对局的数据。
- 本期不支持对 `finished` 对局“重新开启”，仅支持 `running` 对局延长。
- 本期不做复杂审批流或多级权限系统，默认沿用现有管理员能力。

## 3. 当前实现现状
### 3.1 已有基础
- `game_runs` 已包含 `duration_days` 字段，用于记录对局持续天数。
- 对局结束时间按 `created_at + duration_days` 计算。
- 创建对局接口已支持传入 `duration_days`。
- 前端已存在管理员专属页面 `买家池总览`，说明当前系统已经有“管理员菜单 + 管理接口 + 管理页面”这条实现路径。

### 3.2 当前前端管理入口现状
- 当前管理员页不是独立后台，而是挂在工作台左侧菜单体系内。
- 已有管理员页面为 `买家池总览`，说明后续新增“对局管理”入口时，可以直接沿用现有结构，不需要新建一套独立 admin 系统。
- 当前管理员菜单入口是条件显示的，只有满足前端判断条件的账号才能看到。
- 后端管理员接口权限按 `super_admin` 控制，但前端入口判断目前还带有额外用户名限制，这意味着前后端权限口径暂时并不完全一致。

### 3.3 当前问题
- 前端仍存在“1局=7天”的固定常量和固定文案。
- 如果仅在数据库或后端修改 `duration_days`，前端倒计时和展示可能仍按固定 7 天计算。
- 当前虽然已有管理员页面，但还没有“对局管理/时长管理”入口，无法直接承载运行中对局延长操作。
- 因此“管理员延长对局”不能只做后端改值，必须同步梳理前端展示、菜单入口和时间映射逻辑。

## 4. 核心设计
### 4.1 设计原则
- 单一事实来源以当前对局 `run.duration_days` 为准，不再依赖前端写死 `7天`。
- 管理员延长的是“对局总时长”，不是直接改动游戏内资金、订单、库存等经营数据。
- 所有剩余时长展示、封盘判定、阶段页只读边界，都应以最新 `duration_days` 重新计算。

### 4.2 操作定义
管理员可对 `running` 状态对局执行如下操作：
- 延长 7 天
- 延长 14 天
- 自定义延长 N 天

建议最终落库方式：
- `duration_days = duration_days + extend_days`

而不是直接覆盖为固定值。这样更符合“续期”的业务语义。

### 4.3 状态约束
- 仅允许 `run.status = running` 的对局执行延长。
- `finished` / `abandoned` 状态禁止延长。
- 若对局已达到结束边界但状态尚未自动收口，需要先统一判定是否仍属于可操作窗口。

推荐规则：
- 若当前游戏时钟已达到或超过结束边界，则优先收口为 `finished`，不允许再延长。
- 若仍在结束边界之前，则允许延长。

这样可以避免“已经结束又被拉回运行中”的状态歧义。

## 5. 后端方案
### 5.1 管理接口建议
新增管理员接口，例如：
- `POST /game/admin/runs/{run_id}/extend`

请求参数建议：
```json
{
  "extend_days": 7
}
```

返回建议包含：
- `run_id`
- `old_duration_days`
- `new_duration_days`
- `new_end_time`
- `status`

### 5.2 后端校验规则
- `extend_days` 必须为正整数。
- `run_id` 必须存在。
- 当前 run 必须属于可管理范围。
- 当前 run 状态必须为 `running`。
- 若当前时钟已达到结束边界，则拒绝延长。

### 5.3 结束判定兼容
现有结束判定继续保持：
- `run_end_time = run.created_at + timedelta(days=run.duration_days)`

延长后无需新增第二套结束逻辑，只要 `duration_days` 更新成功，结束时间自然顺延。

### 5.4 缓存与读接口
若当前存在对局概览、历史总结、工作台统计缓存，需要在延长成功后清理相关缓存，避免出现：
- 后端已延长
- 页面仍显示旧剩余时长/旧结束时间

建议延长成功后清理与该 `run_id` 相关的概览缓存和历史摘要缓存。

## 6. 前端方案
### 6.1 入口形态建议
建议不新建独立后台，而是沿用当前工作台左侧菜单里的管理员入口模式，在现有 `买家池总览` 同级新增一个菜单项：
- 推荐名称：`对局管理`
- 备选名称：`对局时长管理`
- 不建议直接命名为 `延长对局`，因为后续该页面还可以继续承载“强制结束对局”“查看运行中对局”“运营干预日志”等能力。

推荐原因：
- 改动范围最小，符合现有前端结构。
- 与现有管理员页面组织方式一致，用户认知成本低。
- 便于后续逐步扩展更多管理员能力。

### 6.2 页面内容建议
`对局管理` 页面建议包含：
- 运行中对局列表
- 玩家信息（用户名/姓名）
- 对局编号
- 当前状态
- 当前持续天数
- 预计结束时间
- 快捷操作：`延长7天`、`延长14天`
- 自定义延长输入框
- 最近一次延长结果提示

建议展示二次确认信息：
- 当前对局编号
- 当前持续天数
- 延长后持续天数
- 延长后预计结束时间

### 6.3 菜单与权限显示建议
- 左侧菜单项仅对管理员展示。
- 前端管理员入口判断建议统一收口到 `role=super_admin`，不要额外依赖固定用户名。
- 若继续保留用户名白名单判断，会造成“后端有权限、前端没入口”的体验不一致。

### 6.4 工作台展示调整
前端不再写死“1局=7天”，而是改为读取当前对局的真实时长信息。

重点包括：
- 总倒计时
- 当前游戏日展示
- 周期说明文案
- 历史经营记录中的周期字段
- 管理员查看指定对局时的倒计时说明

### 6.5 时间映射处理
前端现状中存在固定总时长与固定说明文案，因此要统一改成基于当前 run 动态计算。

至少需要确认：
- 剩余时间是否按当前 run 的 `duration_days` 重新换算
- 页面中的“第几天 / 剩余多少游戏天”是否同步刷新
- 历史回溯页展示是否仍沿用历史 run 的实际时长
- 管理员页面切换不同 run 时，右侧倒计时信息是否同步更新

### 6.6 最小可行版本（MVP）
第一期建议只做最小可行范围：
- 左侧菜单新增 `对局管理`
- 页面只展示运行中对局
- 支持 `延长7天` / `延长14天` / 自定义延长
- 延长成功后刷新当前列表与当前对局剩余时长展示

第一期暂不纳入：
- 强制结束对局
- 批量延长
- 完整操作日志页
- 更复杂的运营筛选器

这样可以先把“管理员手动续期”主链路跑通，再逐步扩展。
## 7. 风险点
### 7.1 最大风险：前后端时间口径不一致
如果只改后端 `duration_days`，而前端继续按固定 7 天计算，会出现：
- 页面提示即将结束，但后端实际未结束
- 页面显示剩余时间不准确
- 玩家对延长是否成功产生疑惑

### 7.2 状态边界风险
若一个对局已经理论结束，但状态尚未及时更新，此时管理员点击延长，可能出现边界歧义。

建议统一按“先判定是否已结束，再决定是否可延长”处理。

### 7.3 漏改风险
如果有其他模块也依赖固定 7 天常量，而本次只改工作台页，可能导致：
- 某些阶段页展示仍按旧规则
- 管理页和玩家页表现不一致

### 7.4 审计风险
管理员延长对局属于运营干预动作，建议至少保留操作日志，记录：
- 操作人
- 对局 ID
- 原时长
- 延长天数
- 新时长
- 操作时间

## 8. 推荐实施顺序
1. 梳理所有依赖固定 `7天` / 固定总时长的前端展示和计算点，并确认当前管理员菜单挂载点与路由扩展方式。
2. 统一管理员入口权限口径，前端入口显示规则与后端 `super_admin` 权限保持一致。
3. 新增管理员延长接口，并只允许 `running` 对局调用。
4. 接口返回最新 `duration_days` 与新的结束时间。
5. 前端左侧菜单新增 `对局管理` 入口，并完成运行中对局列表与延长操作页。
6. 工作台、历史经营记录、相关阶段页统一改为基于当前 run 动态展示时长。
7. 清理相关缓存并补充延长后的联调验证。
8. 增加操作日志与异常场景校验。

## 9. 权限与菜单口径建议
- 后端权限校验以 `role=super_admin` 为准。
- 前端菜单显示条件建议与后端保持一致，不再额外绑定固定用户名。
- 管理员相关页面统一放在现有工作台管理员入口体系内，避免形成“半套独立后台、半套嵌入工作台”的混合结构。
- 若后续新增更多管理员页面，建议统一归档到 `admin` 模块并保持同级菜单扩展。

## 10. 验收标准
### 10.1 功能验收
- 管理员可在左侧菜单看到 `对局管理` 入口并成功进入页面。
- 管理员可成功对运行中的对局执行延长操作。
- 延长后，对局新的结束时间按最新 `duration_days` 生效。
- 玩家工作台看到的总倒计时、周期文案、历史记录周期字段与最新时长一致。
- 已结束对局不能被延长。
- 延长操作不会修改资金、订单、库存等经营数据。

### 10.2 回归验收
- 未被延长的对局行为保持不变。
- 历史对局回溯能力不受影响。
- 到期自动封盘能力不受影响。
- Shopee 订单相关查询/演化逻辑在延长前后保持一致，不出现提前封盘或延迟封盘异常。
- 现有 `买家池总览` 管理页面入口与行为不受影响。

## 11. 结论与建议
如果目标是“最强运营能力”，管理员延长运行中对局是可行且有价值的方案。

推荐落地原则：
- 后端以 `duration_days` 作为唯一真实来源。
- 前端全面去掉固定 `7天` 假设，改为读取当前 run 实际时长。
- 管理入口沿用现有左侧菜单管理员模式，在 `买家池总览` 同级新增 `对局管理`。
- 前后端权限口径统一到 `super_admin`，避免出现有权限但无入口的情况。
- 已结束对局不允许重开，避免状态语义混乱。

这样既能满足运营临时续期诉求，也能为后续扩展“后台可选 7/14/30 天”保留统一实现基础。

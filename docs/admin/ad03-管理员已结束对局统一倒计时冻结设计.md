# ad03-管理员已结束对局统一倒计时冻结设计

## 1. 目标
- 当管理员在 `买家池总览` 或 `对局管理` 中选中某个对局时，右侧统一时间中心必须严格显示该对局自己的时间信息。
- 当选中对局已结束（`finished`）时，统一倒计时、当前游戏日、游戏时刻都应冻结在该对局截止时刻，不再继续跟随当前真实时间跳动。
- 前后端对“结束态时间”语义保持一致，避免出现“Day 已到终点但时钟还在跳”或“已 finished 但顶部倒计时仍在走”的混乱表现。

## 2. 场景说明
### 2.1 适用场景
- 管理员在 `买家池总览` 中切换查看不同对局时，需要右侧时间中心同步切换为对应对局的时间语义。
- 管理员选择已结束对局时，需要看到该局的最终截止时间，而不是当前系统时间推演出的实时倒计时。
- 教学、演示、回放场景中，需要用统一时间中心准确表达“这局已经停在什么时候结束”。

### 2.2 非目标
- 本期不重做整套时间系统，只修正管理员选中对局时统一时间中心的结束态语义。
- 本期不改变 `finished` 对局的历史回溯入口逻辑。
- 本期不引入新的时间配置中心或新的全局时钟模型。

## 3. 当前问题现状
### 3.1 前端现状
管理员页会把选中对局的上下文传给工作台右侧时间中心，包括：
- `runId`
- `status`
- `createdAt`
- `durationDays`
- `manualEndTime`
- `endTime`
- `gameClock`

但右侧时间中心当前仍直接用：
- `elapsedSeconds = nowMs - createdAt`
- `remainSeconds = totalRealSeconds - elapsedSeconds`

因此只要页面还开着，顶部统一倒计时就会继续变化，即使该局已经 `finished`。

### 3.2 后端现状
管理员买家池总览接口已经返回：
- `selected_run_status`
- `selected_run_manual_end_time`
- `selected_run_end_time`
- `game_clock`

但 `game_clock` 当前仍按“当前时间 now”参与计算，没有在 finished 时截断到该局的结束时间，因此：
- `selected_run_day_index` 可能已经停在最终天
- `game_clock` 仍会继续跳动

### 3.3 用户感知问题
当前会出现以下体验错误：
- 管理员选中已结束对局，但右侧顶部倒计时还在持续减少。
- 右侧蓝色游戏时刻还在变化，像是该局仍在运行。
- 同一块区域里“状态是 finished”但“时钟却仍在动”，语义自相矛盾。

## 4. 核心设计原则
### 4.1 选中哪个对局，就显示哪个对局的时间
统一时间中心在管理员模式下必须以“当前选中对局”为唯一来源，不混用当前玩家运行中对局或系统当前时钟。

### 4.2 已结束对局必须冻结在结束时刻
一旦对局状态为 `finished`，时间展示应停在该对局自己的结束截点：
- 倒计时不再变化
- 游戏日不再推进
- 游戏时刻不再跳动

### 4.3 后端定义语义，前端负责正确呈现
- 后端负责定义该对局“此刻应该显示的游戏时间”。
- 前端负责根据状态决定展示“倒计时”还是“已结束”。
- 前后端必须共享同一个结束截点，不能分别按不同时间源推导。

## 5. 结束时间语义
### 5.1 对局有效结束时间
统一定义：
- 若 `manual_end_time` 存在，则该局有效结束时间为 `manual_end_time`
- 否则为 `created_at + duration_days`

即：
- `effective_end_time = manual_end_time ?? (created_at + duration_days)`

### 5.2 elapsed 计算规则
建议所有管理员选中对局的时间推导统一改为基于 `effective_compare_time`：

- `running`：`effective_compare_time = min(now, effective_end_time)`
- `finished`：`effective_compare_time = effective_end_time`

然后统一计算：
- `elapsed = effective_compare_time - created_at`

这样可以保证：
- 运行中对局正常推进
- 已结束对局完全冻结
- 即使结束后页面继续打开，时间也不会继续漂移

### 5.3 remain 计算规则
- `running`：`remain = max(0, effective_end_time - now)`
- `finished`：`remain = 0`

因此 finished 对局的顶部总倒计时不应继续显示递减值。

## 6. 后端方案
### 6.1 修改目标
后端重点修正管理员买家池总览接口生成的：
- `selected_run_day_index`
- `game_clock`
- `game_hour`
- `game_minute`

使其在 finished 对局下统一按 `effective_end_time` 截断。

### 6.2 建议修改点
建议收口到对局 elapsed 计算逻辑，而不是只在某一个接口单独特判。

推荐优先调整：
- `_resolve_run_elapsed_seconds(run, now=None)`

建议行为：
1. 先算出 `run_end_time`
2. 再基于状态和结束时间确定 `effective_compare_time`
3. 最终统一返回 `created_at -> effective_compare_time` 的秒数

这样可联动修正：
- `_resolve_game_day_float_by_run`
- `_resolve_game_day_index_by_run`
- `_resolve_game_clock_info_by_run`

### 6.3 接口输出要求
管理员买家池总览接口应满足：
- 若 selected run 为 `finished`，则每次刷新返回的 `game_clock` 不再变化
- 若 selected run 为 `finished`，则 `selected_run_day_index` 固定为最终日
- 若 selected run 为 `finished`，则返回的 `selected_run_end_time` 与游戏时钟语义一致

### 6.4 缓存要求
当前接口存在 overview 缓存，因此修改后需确认：
- `finished` 对局缓存不会引入新的跳动问题
- 若时间冻结后缓存命中，返回的 `game_clock` 应稳定不变
- 对运行中对局仍维持原有概览缓存节奏

## 7. 前端方案
### 7.1 顶部倒计时卡片
管理员模式下若右侧使用的是选中对局上下文，则：
- `running`：按当前对局剩余时间显示实时倒计时
- `finished`：不再继续倒计时，建议直接显示 `已结束`

不建议 finished 时继续显示一个不断变化的秒表，也不建议继续按本地 `nowMs - createdAt` 推导。

### 7.2 当前游戏日卡片
- `running`：按当前对局实时推进显示
- `finished`：固定显示最终 `Day`

副文案建议保留：
- `对局 #id · 玩家 xxx · finished`

### 7.3 游戏日历卡片
蓝色 `Game Time` 建议继续优先显示后端返回的 `gameClock`，但前提是后端语义已修正。

最终要求：
- finished 时 `gameClock` 固定不变
- 日历高亮保持在最终游戏日位置

### 7.4 UI 文案建议
对 finished 对局建议把顶部标题从：
- `选中对局倒计时`

调整为条件化文案：
- `running`：`选中对局倒计时`
- `finished`：`选中对局已结束`

展示值建议：
- `running`：`3天 12小时 08分钟 11秒`
- `finished`：`已结束`

辅助文案可显示：
- `截止于 Day N / YYYY-MM-DD HH:mm:ss`

这样比单纯显示 `00天00小时00分00秒` 更符合业务语义。

## 8. 前后端职责边界
| 层 | 负责内容 |
|---|---|
| 后端 | 定义选中对局当前应显示的游戏时间、游戏日、结束截点 |
| 前端 | 基于状态决定显示倒计时还是已结束，并渲染最终值 |
| 前后端共同 | 保证 `day / game_clock / remain` 三者语义一致 |

## 9. 风险点
### 9.1 只修前端的风险
如果只让前端在 `finished` 时显示 `已结束`，但后端 `game_clock` 仍继续变化，则：
- 顶部卡片看起来正常
- 但游戏时刻仍会跳动
- 用户仍会感知到不一致

### 9.2 只修后端的风险
如果只修后端 `game_clock`，但前端顶部倒计时仍按 `nowMs - createdAt` 算，则：
- 蓝色时钟会稳定
- 但顶部总倒计时仍继续减少

### 9.3 最佳方案
最佳方案是：
1. 后端统一封口时间语义
2. 前端按 finished 状态切换展示形态
3. 联调时同时验证倒计时、游戏日、游戏时刻三者一致

## 10. 推荐实施顺序
1. 后端梳理 `effective_end_time` 与 elapsed 截断规则。
2. 修改管理员买家池总览接口相关时间计算，使 finished 对局 `game_clock` 固定。
3. 前端统一时间中心增加 finished 展示分支：顶部卡片显示 `已结束`，剩余时间归零。
4. 联调管理员 `买家池总览` 与 `对局管理`，验证切换不同对局时右侧时间中心同步正确。
5. 补充回归验证，确认 running 对局实时倒计时能力不受影响。

## 11. 验收标准
### 11.1 功能验收
- 管理员选中任意运行中对局时，右侧统一时间中心显示该对局实时倒计时。
- 管理员选中任意已结束对局时，右侧顶部卡片不再继续跳动。
- 已结束对局的 `当前游戏日` 固定为最终日。
- 已结束对局的蓝色 `Game Time` 固定在结束时刻，不再变化。
- 同一已结束对局连续刷新页面或等待数分钟后，时间展示保持一致。

### 11.2 回归验收
- 未结束的 running 对局倒计时仍正常推进。
- 历史对局回溯页不受影响。
- 对局管理页、买家池总览页切换 run 时，右侧时间中心仍能同步切换到选中对局。
- 现有对局延长、续期恢复、游戏年限追加逻辑不受影响。

## 12. 结论
该问题的本质不是单点 UI bug，而是“结束态时间语义”在前后端没有统一封口。

推荐最终落地原则：
- 选中哪个对局，就展示哪个对局的时间
- 已结束对局，统一冻结在该局自己的结束截点
- 后端负责定义冻结后的最终时钟，前端负责把 finished 明确呈现为“已结束”而不是继续倒计时

按这个方案落地后，管理员视角下的统一时间中心才能真正具备可靠的时间语义。

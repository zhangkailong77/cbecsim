# 订单自动模拟被未来 tick_time 脏数据卡死问题

## 问题概述

在 Shopee 订单页面持续刷新 `run_id=6` 的待发货列表时，前端请求能够稳定命中后端接口 `GET /shopee/runs/6/orders`，但最近 1~2 小时数据库表 `shopee_order_generation_log` 没有新增记录，页面体感为“订单下单模拟没有继续推进”。

该问题最终定位为：

- 订单自动模拟函数 `_auto_simulate_orders_by_game_hour` 会读取 `shopee_order_generation_log.tick_time` 的最大值作为 `base_tick`
- 历史脏数据中存在一条未来游戏时间的 `tick_time=2027-05-08 08:59:42`
- 当前重新计算出的 `current_game_tick=2026-05-01 07:59:42`
- 导致 `missing_steps` 为负数，自动模拟函数直接 return，不再补跑任何 tick，也不会写入新的生成日志

简而言之：**历史未来 tick 脏数据把自动模拟基准时间顶到了未来，导致 run 6 今天开始“永远不再自动下单”。**

---

## 影响范围

### 直接影响

- Shopee 订单自动模拟无法继续推进
- `shopee_order_generation_log` 不再新增当天记录
- 页面刷新虽然命中 `/shopee/runs/{run_id}/orders`，但不会产生新的模拟订单
- 用户体感为“等了很久也没有新订单”

### 间接影响

- 由于没有新的自动模拟日志，容易误判为前端未请求、概率过低、商品/买家条件不足
- 同类历史脏数据若出现在其他 run，也会导致相同卡死问题

---

## 复现现象

### 页面侧现象

- 刷新 Shopee 订单页面时，后端 access log 正常出现：

```text
GET /shopee/runs/6/orders?type=toship&source=to_process&sort_by=ship_by_date_asc&page=1&page_size=20 200 OK
```

说明前端请求确实到达了正确后端与正确 run。

### 数据库侧现象

查询 `shopee_order_generation_log` 最近 1~2 小时记录为空：

```sql
SELECT
  id,
  run_id,
  tick_time,
  active_buyer_count,
  candidate_product_count,
  generated_order_count,
  skip_reasons_json,
  created_at
FROM shopee_order_generation_log
WHERE run_id = 6
  AND created_at >= NOW() - INTERVAL 2 HOUR
ORDER BY created_at DESC, id DESC;
```

说明今天最近 2 小时没有新的自动模拟日志写入。

### 调试日志证据

在 `_auto_simulate_orders_by_game_hour` 中加入调试日志后，刷新页面得到：

```text
[order-auto-sim] run_id=6 user_id=3 latest_tick_time=2027-05-08 08:59:42 base_tick=2027-05-08 08:59:42 current_game_tick=2026-05-01 07:59:42 step_seconds=600 missing_steps=-53574 max_ticks_per_request=1
```

该日志直接证明：

- `latest_tick_time` 来自历史脏数据，落在未来游戏时刻
- `current_game_tick` 比 `base_tick` 更早
- `missing_steps` 为负数
- 自动模拟提前 return

---

## 根因分析

问题代码位于：

- `backend/apps/api-gateway/app/api/routes/shopee.py`
- 函数：`_auto_simulate_orders_by_game_hour`

关键逻辑为：

```python
latest_tick_time = (
    db.query(func.max(ShopeeOrderGenerationLog.tick_time))
    .filter(
        ShopeeOrderGenerationLog.run_id == run.id,
        ShopeeOrderGenerationLog.user_id == user_id,
    )
    .scalar()
)
base_tick = latest_tick_time or run.created_at
current_game_tick = _resolve_game_hour_tick_by_run(run)
step_seconds = max(1, int(REAL_SECONDS_PER_GAME_HOUR * ORDER_SIM_TICK_GAME_HOURS))
missing_steps = int((current_game_tick - base_tick).total_seconds() // step_seconds)
if missing_steps <= 0:
    return
```

### 根因总结

1. 代码默认信任 `latest_tick_time` 一定不晚于当前游戏时刻
2. 历史错误逻辑曾写入了未来 `tick_time`
3. 当前代码没有对 `latest_tick_time > current_game_tick` 做纠偏
4. 导致 `missing_steps <= 0` 永久成立
5. 自动模拟彻底卡死

---

## 为什么不是其他问题

### 不是前端没请求

因为 access log 明确显示 `/shopee/runs/6/orders` 被重复请求。

### 不是概率过低

如果只是概率过低，仍然会产生新的 `shopee_order_generation_log`，只是 `generated_order_count=0`。

当前是最近 1~2 小时完全没有新日志，说明问题发生在“写模拟日志之前”。

### 不是商品/买家为空

同理，如果是 `no_live_products`、`no_stock`、`no_active_buyer_profiles`，也应出现新的日志记录并写入 `skip_reasons_json`。

当前没有任何新日志，说明模拟根本没有进入执行阶段。

---

## 修复建议

### 代码修复（推荐）

在 `_auto_simulate_orders_by_game_hour` 中增加基准时间纠偏：

```python
current_game_tick = _resolve_game_hour_tick_by_run(run)
base_tick = latest_tick_time or run.created_at
if base_tick and base_tick > current_game_tick:
    base_tick = run.created_at
```

更稳妥的版本可同时打印 warning，便于后续追踪历史脏数据：

```python
if base_tick and base_tick > current_game_tick:
    logger.warning(
        "[order-auto-sim] reset future base_tick run_id=%s user_id=%s base_tick=%s current_game_tick=%s",
        run.id,
        user_id,
        base_tick,
        current_game_tick,
    )
    base_tick = run.created_at
```

### 数据修复（可选）

若希望 run 6 立即恢复，可对历史未来 tick 记录进行人工回填或删除。例如将该 run 最近一条异常 `tick_time` 改回不晚于当前游戏时刻的值。

但**仅修数据库不修代码**会留下同类隐患，因此建议代码修复优先。

---

## 验证方案

修复后，应验证：

1. 刷新 `/shopee/runs/6/orders` 时，调试日志中的 `missing_steps` 不再为负数
2. `shopee_order_generation_log` 最近 1 小时开始出现新记录
3. 若概率未命中，`generated_order_count=0` 但仍应有新日志
4. 若概率命中，应能看到新的 `SIM...` 订单号产生

建议使用以下 SQL 复查：

```sql
SELECT
  id,
  run_id,
  tick_time,
  active_buyer_count,
  candidate_product_count,
  generated_order_count,
  skip_reasons_json,
  created_at
FROM shopee_order_generation_log
WHERE run_id = 6
  AND created_at >= NOW() - INTERVAL 1 HOUR
ORDER BY created_at DESC, id DESC;
```

## 结论

这是一个**历史未来 tick 脏数据 + 当前缺少基准时间纠偏保护**共同导致的自动模拟卡死问题。

只要 `latest_tick_time` 晚于 `current_game_tick`，该 run 就会持续卡在 `missing_steps <= 0`，页面刷新再多次也不会新增订单模拟日志。

建议尽快补上代码防护，并视需要清理历史异常 tick 数据。
